----------------------------------------------------------------
1. CLAUDE.md가 특별한 이유
----------------------------------------------------------------
Claude Code는 대화 시작 시 프로젝트 안의 CLAUDE.md만 자동으로 읽음.
다른 이름의 .md 파일은 무시됨.

자동화 규칙, 명령어 등을 Claude에게 기억시키려면
반드시 CLAUDE.md 안에 있어야 함.

----------------------------------------------------------------
2. @파일명으로 파일 분리하는 방법
----------------------------------------------------------------
CLAUDE.md 안에서 @파일명 을 쓰면
Claude가 그 파일도 이어서 읽음.

  CLAUDE.md 내용:
  ────────────────────
  @claude_css_folder.md
  @claude_til.md
  ────────────────────

Claude 입장에서는 세 파일을 합친 것과 동일하게 인식.
@는 "이 자리에 저 파일 내용을 붙여넣어"와 같은 동작.
import 같은 별도 키워드 없이 @ 뒤에 파일 경로만 쓰면 됨.

덕분에 CLAUDE.md 하나에 다 쌓지 않고
기능별로 파일을 나눠도 Claude는 전부 인식함.

----------------------------------------------------------------
3. 현재 적용된 구조
----------------------------------------------------------------
hancom/hancom/
├── CLAUDE.md              ← 목차만 (@파일명 나열)
├── claude_css_folder.md   ← CSS 폴더 생성 자동화
└── claude_til.md          ← TIL 저장 자동화

새 자동화 규칙 추가 시:
  1. claude_xxx.md 파일 새로 작성
  2. CLAUDE.md에 @claude_xxx.md 한 줄 추가

----------------------------------------------------------------
4. TIL 저장 자동화 규칙 (claude_til.md 내용 요약)
----------------------------------------------------------------
트리거: "TIL 저장해줘" / "오늘 내용 정리해줘" / "블로그 정리해줘"
저장 경로: hancom/hancom/TIL_blog/
파일명 형식: YYMMDD_과목_번호_주제.txt
  예시: 260702_css_12_neon.txt
       260702_claude_01_claudemd_구조.txt

과목 구분:
  css    → CSS 강의 관련
  claude → Claude 사용법 관련
  (이후 필요에 따라 추가)

[수정: 2026-07-02] 5. .claude/settings 파일 이해
----------------------------------------------------------------
Claude Code가 프로젝트 첫 실행 시 .claude/ 폴더와 두 파일을 자동 생성.
삭제해도 재실행 시 다시 생성됨 (단, settings.local.json의 항상 허용 목록은 초기화).

settings.json
  - 팀 공유용, git에 올려도 됨
  - 기본값은 빈 객체 {}
  - 프로젝트 공통 설정을 넣는 용도

settings.local.json
  - 로컬 전용, git에 올리면 안 됨
  - Claude 명령어 실행 시 "항상 허용"을 누른 명령어들이 누적 저장
  - "허용" → 그 번만 실행 / "항상 허용" → 이 파일에 저장 → 이후 질문 없이 실행

----------------------------------------------------------------
[수정: 2026-07-08] 6. 권한(permissions.allow) 규칙 직접 추가하기
----------------------------------------------------------------
settings.json의 permissions.allow 배열에 규칙을 직접 추가하면
해당 도구/명령어를 매번 승인 없이 자동 실행시킬 수 있음.

규칙 문법:
  "PowerShell(New-Item *)"  → New-Item으로 시작하는 PowerShell 명령만 허용
  "Bash(mkdir *)"           → mkdir로 시작하는 Bash 명령만 허용
  "Write"                   → Write 도구 전체를 조건 없이 허용
  "Edit"                    → Edit 도구 전체를 조건 없이 허용

도구 이름만 쓰면(Write, Edit처럼) 그 도구의 모든 호출이 허용됨.
괄호 안에 패턴을 쓰면(PowerShell(New-Item *)) 그 패턴에 맞는 호출만 허용됨.

Write 도구 vs Edit 도구 차이:
  Write → 파일을 통째로 새로 씀 (신규 생성 / 전체 덮어쓰기)
  Edit  → 기존 파일 안에서 문자열 일부만 찾아 바꿈 (부분 수정)

----------------------------------------------------------------
[수정: 2026-07-08] 7. 프로젝트 settings.json vs 전역 settings.json
----------------------------------------------------------------
같은 이름의 파일이지만 위치에 따라 적용 범위가 다름.

  c:\...\hancom\.claude\settings.json   → 이 프로젝트에서만 적용 (git 공유)
  C:\Users\...\.claude\settings.json    → 사용자 전역, 모든 프로젝트에 적용

전역에 Write/Edit 자동 허용 같은 규칙을 걸면 편리하지만,
다른(특히 중요한) 프로젝트를 열었을 때도 검토 없이 파일이 써지고 수정됨.
→ 위험도가 낮은 연습용 프로젝트는 프로젝트 단위로, 실무/중요 프로젝트는
   신중하게 개별 판단하는 것이 안전함.

[수정: 2026-07-08] 8. 하네스 엔지니어링 / 루프 엔지니어링 / 멀티에이전트
----------------------------------------------------------------
하네스(harness) 엔지니어링
  AI 모델 하나를 감싸는 도구·권한·컨텍스트 관리 시스템을 설계하는 것.
  settings.json의 permission 규칙, 어떤 도구를 줄지, 컨텍스트를
  어떻게 압축할지 등이 전부 "하네스"에 해당함. Claude Code 자체가 하나의 하네스.

루프(loop) 엔지니어링
  한 에이전트가 "실행 → 결과 확인 → 수정 → 재실행"을 반복하도록
  설계하는 것. 서브에이전트가 여러 개일 필요는 없음 (/loop 스킬이 예시).

멀티에이전트 오케스트레이션은 이 둘 위에 얹을 수 있는 응용 패턴 중 하나.

정적 allow 목록 대신 "맥락에 맞게 판단해서 승인"하게 만드는 방법:
  1. Auto mode
     - 매 액션을 작은 분류 모델이 맥락(명령 종류, 파일, 되돌릴 수 있는지)을
       보고 즉석에서 허용/거부/확인을 판단
     - settings.json의 autoMode(allow/soft_deny/hard_deny)로 기준 커스터마이징 가능
  2. PreToolUse 훅 (type: "prompt" 또는 "agent")
     - 도구 호출 전에 별도 LLM/에이전트를 불러 안전 여부를 판단시키고
       그 결과로 allow/deny/ask를 결정

트레이드오프:
  "내가 매번 클릭"하던 위험을 "판단하는 에이전트를 신뢰"하는 위험으로
  옮기는 것뿐. 그 에이전트가 오판하거나 프롬프트 인젝션에 속으면
  위험한 액션도 통과될 수 있음.
  → rm -rf, force push처럼 되돌릴 수 없는 작업은 판단 에이전트도
    못 뒤집는 hard_deny 규칙으로 따로 고정해두는 것이 일반적.
