# [claude 04] groq-chat을 Vercel에 배포하며 실제 겪은 흐름

> 작성: 2026-07-15

## Q&A

**Q. 원래 `hancom` 모노레포 안(`07_groq`)에서 만들던 걸 왜 독립 프로젝트로 옮겼나?**

수업 진도상 처음엔 `hancom/07_groq` 폴더 안에서 만들었지만, 이걸 Vercel에 배포하려면 **모노레포 전체가 아니라 이 프로젝트 하나만 담은 별도 GitHub 저장소**가 있어야 했다. 그래서 코드를 바탕화면 `groq-chat` 폴더로 옮기고, 모노레포 쪽에선 `07_groq`를 삭제했다.

**Q. 분리한 뒤 `git push`를 했는데 뭔가 이상했다. 원격 저장소 연결을 두 번 해야 했던 이유는?**

폴더를 옮긴 뒤 처음 `git push`를 했을 때, **연결된 원격 저장소(origin)가 여전히 기존 `hancom` 레포를 가리키고 있었다.** 폴더는 옮겼지만 git 설정(`.git` 안의 origin 주소)까지 완전히 새로 정리되지 않았던 것이다. 그래서 새 GitHub 저장소(`naemnaem99/groq-chat`)를 만들고, origin을 그쪽으로 다시 연결(또는 새로 `git init`)한 뒤에야 제대로 push가 됐다.

> 정확히는 "깃 연결이 하나의 레포에만 되는" 게 아니라, `origin`이라는 이름표가 **한 번에 하나의 주소만** 가리킬 수 있는 것이다. 여러 원격 저장소를 동시에 등록하는 것도 가능하지만(`git remote add 다른이름 주소`), 보통은 `origin` 하나만 쓰기 때문에 폴더를 복사해왔을 때 "예전 주소가 그대로 남아있는" 문제가 생기기 쉽다. `git remote set-url origin 새주소`로 바꾸거나, 아예 새로 `git init`하면 해결된다.

**Q. 로컬에서는 Express 서버(`server/index.js`)로 잘 돌아갔는데, Vercel에 배포할 땐 왜 코드를 또 손봐야 했나?**

로컬 서버는 "항상 켜져 있는 프로그램"이 요청을 받는 방식이다. 그런데 Vercel은 그런 상시 서버를 그대로 못 돌리고, 대신 **`api/` 폴더 안의 파일 하나하나를 "요청이 올 때만 잠깐 실행되는 서버리스 함수"로 인식**하는 구조다. 그래서 기존 Express 서버가 하던 일(Groq API에 대신 요청 보내기)을 `api/chat.js`라는 서버리스 함수 형태로 새로 옮겨 적어야 했다.

**Q. 이 과정에서 API 키를 정확히 어떻게 다룬 건가?**

- 로컬 테스트 때는 `server/.env` 파일에 `GROQ_API_KEY=실제키값`을 적어두고, Express 서버가 `process.env.GROQ_API_KEY`로 그 값을 읽어서 Groq API를 대신 호출했다.
- `.env` 파일은 `.gitignore`에 등록돼 있어서 **GitHub에는 절대 올라가지 않는다** (`git ls-files`로 직접 확인해봐도 추적되지 않음). 즉 Vercel이 GitHub 저장소를 보고 배포해도, 그 안에는 키 값이 아예 없다.
- 그래서 **Vercel 대시보드(프로젝트 설정 → Environment Variables)에 `GROQ_API_KEY`를 직접 입력해서 등록**해야 했다. 이렇게 등록해두면 Vercel이 `api/chat.js`를 실행할 때 그 값을 `process.env.GROQ_API_KEY`로 자동으로 넣어준다 — 로컬의 `.env`와 똑같은 역할을 Vercel 쪽에서는 "환경변수 설정"이 대신하는 것이다.
- 결과적으로 키는 **① 로컬 `.env`(로컬 테스트용) ② Vercel 환경변수(배포용)** 두 군데에만 존재하고, **깃 저장소 안에도 브라우저로 전달되는 코드 안에도 절대 들어가지 않는다.**

**Q. 이후 "디자인 수정하고 푸시해줘"라고 하니 실제로 배포까지 됐다. 이게 어떻게 가능한가?**

GitHub 저장소와 Vercel 프로젝트를 한 번 연결해두면(Git Integration), 그 뒤로는 별도의 "배포" 행위가 필요 없다.

```
Claude가 로컬 코드 수정
   ↓
git commit
   ↓
git push origin main
   ↓ (GitHub에 push된 걸 Vercel이 감지)
Vercel이 자동으로 새 버전 빌드 + 배포
   ↓
라이브 URL이 몇 초 안에 새 코드로 교체됨
```

Claude가 Vercel에 직접 API 요청을 보내는 게 아니라, **평소처럼 git push만 하면** Vercel이 알아서 감지해서 재배포한다.

## 개념 요약

### 전체 흐름 한눈에

| 단계 | 내용 |
|---|---|
| 1 | 모노레포 안(`07_groq`)에서 먼저 작업 |
| 2 | 독립 폴더(`groq-chat`)로 분리, 모노레포에서는 삭제 |
| 3 | `git push` 시 origin이 예전 레포를 가리켜서 실패 → 새 GitHub 레포 만들고 origin 재연결 |
| 4 | 커밋 후 Vercel과 연결 (`.vercel/project.json` 생성) |
| 5 | 로컬 Express 서버 방식 → Vercel 서버리스 함수(`api/chat.js`) 방식으로 코드 전환 |
| 6 | API 키를 Vercel 환경변수로 등록 |
| 7 | 이후로는 커밋 + 푸시만 하면 자동 재배포 |

### `origin`은 한 번에 하나의 주소만 가리킨다

**실무 예시**: 기존 프로젝트를 복사해서 새 프로젝트를 시작할 때, `.git` 폴더까지 같이 복사해오면 `origin`이 원본 저장소를 계속 가리키고 있어서 엉뚱한 곳에 push될 뻔하는 실수가 흔하다. 새 프로젝트를 시작할 땐 `git remote -v`로 현재 연결을 먼저 확인하는 습관이 필요하다.

### 로컬 서버 vs 서버리스 함수

| | 로컬 Express 서버 | Vercel 서버리스 함수 |
|---|---|---|
| 실행 방식 | 상시 켜져 있음 | 요청 올 때만 잠깐 실행 |
| 파일 위치 | 자유(`server/index.js` 등) | `api/` 폴더 규칙을 따름 |
| API 키 위치 | `.env` 파일 | Vercel 환경변수 설정 |

**실무 예시**: 프로토타입은 로컬 Express로 빠르게 테스트하고, 배포 단계에서 서버리스 함수 형태로 옮기는 흐름은 개인 프로젝트/사이드 프로젝트에서 흔한 패턴이다.

### API 키 보호 패턴 (서버리스 함수)

```js
// api/chat.js — Vercel 서버리스 함수
module.exports = async (req, res) => {
  const key = process.env.GROQ_API_KEY // 브라우저가 아니라 서버(Vercel)에서만 읽음
  // ...Groq API 호출...
}
```

**실무 예시**: 프론트엔드 코드에 `fetch('https://api.groq.com/...', { headers: { Authorization: 'Bearer sk-...' }})`처럼 키를 직접 넣으면 브라우저 개발자도구에서 그대로 노출된다. 대신 자기 서버(여기서는 Vercel 서버리스 함수)를 한 겹 두고, 키는 서버 환경변수로만 보관 → 클라이언트는 자기 서버에만 요청 → 서버가 실제 API에 대신 요청하는 구조가 표준적인 키 보호 방식이다.
