# [Claude 03] 요청을 효율적으로 정리해서 던지기

> 작성: 2026-07-08

## Q&A

**Q. feel 프로젝트를 만들면서 요청이 잘게 쪼개져서 반복이 많았는데, 어떤 부분이 정상적인 시행착오이고 어떤 부분이 피할 수 있었던 비효율이었나?**

시각적으로 직접 봐야 판단되는 것(이모지 삽입, 픽셀 패턴 디자인, 애니메이션 느낌)은 미리 다 정해놓고 시작해도 결국 보고 나서 뒤집었을 것들이라 비효율이 아니다. 반면 **구조적 규칙**이나 **핵심 컨셉**을 뒤늦게 알게 되어 같은 실수를 반복한 경우는 피할 수 있었던 비효율이다.

**Q. 구체적으로 어떤 반복이 있었나?**

1. **같은 CSS 버그를 세 번 만남** — `position:fixed`인 드래그 요소는 조상 요소에 `transform`/`perspective`가 걸리면 좌표 기준(containing block)이 깨져버린다. body의 `perspective` → `.cards`의 `transform` → 매번 새 조상에 transform을 걸었다가 똑같은 이유로 요소가 사라지는 버그를 세 번 겪었다.
2. **문구/버튼 텍스트를 한 줄씩 여러 메시지로 나눠서 요청** — 버튼 라벨 3단계, placeholder 3종, 초기 프롬프트가 전부 따로따로 왔다.
3. **핵심 컨셉이 뒤늦게 확정** — 처음엔 기분 카드 4개(감정 도트 패턴)를 만들었다가, 최종적으로는 카드 1개짜리 "바구니" 컨셉으로 완전히 재해석되면서 앞의 작업이 대부분 버려졌다.

**Q. 그럼 앞으로는 어떻게 요청하는 게 효율적인가?**

**큰 구조/상호작용 규칙**은 한 번에 몰아서 요청하고, **색·크기·문구 같은 디테일**은 결과물을 보고 나서 하나씩 다듬는 식으로 나눠서 요청한다.

```
"feel"이라는 감정 배출 웹앱을 만들자. 컨셉: 배출 → 선택 → 소각.

1. input+버튼으로 감정 입력 → 클릭 3번에 걸쳐 문구/버튼 텍스트가 단계적으로 바뀌고
   (초기/좋은감정/나쁜감정), 3번째 클릭에 리셋되면서 화면 중앙에 바구니 등장.
   입력한 텍스트는 눈처럼 화면에 흩날림.
2. 바구니: 위쪽 뚫린 픽셀아트 상자, 자유롭게 드래그 가능(항상 fixed 포지션 유지,
   조상에 transform 금지). 입구(상단중앙 1/3)에 닿는 낙하 텍스트만 선택적으로 캐치.
3. 바구니를 처음 만지면 하단에 불꽃 등장, 바구니를 불로 끌고 가면 소각 후 새 바구니 재생성.
4. 입력 기록은 localStorage에 3개 사이클 단위로 저장, 새로고침 시 마지막 사이클만 재생.
스타일: 미니멀 흑백 픽셀아트.
```

이 정도만 있었어도 최종 결과물의 80%는 훨씬 적은 왕복으로 만들 수 있었을 것이다.

## 개념 요약

| 구분 | 예시 | 처리 방식 |
|---|---|---|
| 시각적 판단이 필요한 것 | 이모지/픽셀 디자인, 애니메이션 느낌 | 만들어보고 반복 수정 — 정상적인 과정 |
| 구조적 규칙 | 드래그 요소는 항상 fixed 유지, 조상에 transform 금지 | 처음에 요구사항으로 명시하면 같은 버그 반복 예방 |
| 반복되는 텍스트/문구 | 버튼 라벨 3단계, placeholder 3종 | 표로 한 번에 정리해서 요청 |
| 핵심 컨셉 | "바구니로 텍스트를 담는다" | 먼저 확정하면 그 이전 단계 작업(카드 4개 등)을 생략 가능 |

**핵심**: 시각적 디테일은 반복이 당연하지만, 구조적 규칙과 핵심 컨셉은 미리 한 번에 정리해서 요청할수록 반복 작업이 크게 줄어든다.
