# [claude 05] 클로드로 웹페이지 만들기 — 단계별로 비교해보며 느낀 것

> 작성: 2026-07-15

같은 "계산기 웹페이지 만들기"를 조건만 바꿔가며 단계별로 시도해본 기록. (`09_claude_web` 폴더: `01_no_plan` ~ `07_haiku`)

## Q&A

**Q. 플랜 없이 그냥 "계산기 만들어줘"라고 하면 어떤가?**

나쁘지 않다. 다만 다른 사람들이 같은 방식으로 만든 결과물과 전반적으로 비슷하다 — Claude가 스스로 판단한 "무난하게 예쁜" 기본값으로 수렴하는 느낌이다.

**Q. 플랜 모드를 쓰면서 클로드가 방향성을 먼저 물어보게 하면 나아지나?**

질문 자체가 꽤 기본적인 수준이라, 결과가 크게 달라지지 않았다. 클로드의 질문에만 의존하기보다, **애초에 방향성을 구체적으로 정해서 지시해야** 할 것 같다.

**Q. `frontend-design` 스킬을 쓰면?**

조금 더 낫다 정도. "한 번에 딸깍"으로 완성품이 나오길 기대할 수는 없다. 결국 **내가 원하는 걸 스스로 정확히 인식하고, 그걸 지시할 수 있어야** 한다. 그런데 아는 게 없으면 애초에 뭘 물어봐야 할지도 감이 안 잡힌다 — 그래서 "아는 게 많아야" 한다는 결론에 이르렀다. 대안으로는 클로드에게 세부적인 질문을 먼저 해달라고 요청하는 방법이 있을 듯하다.

**Q. 다른 사이트의 디자인을 `design.md`로 따와서 참고자료로 주면?**

남이 만든 디자인 양식을 그대로 가져오는 거라 새롭지는 않다. 결국 여러 웹 디자인 사이트를 직접 둘러보며 경험을 쌓아야, 내가 스스로 생각하고 만들 수 있는 범위가 늘어난다. 경험이 쌓이면 클로드가 묻기도 전에 원하는 바를 명확히 지시할 수 있고, 경험이 부족하면 반대로 **클로드가 나조차 모르는 내 니즈를 파고들어 물어보도록 유도**해야 한다.

> 이 "질문을 잘 유도하는 방법"은 아직 답을 못 찾았고, 계속 고민해볼 주제로 남겨둔다.

**Q. 하위 모델(Haiku)로 만들면?**

역시 모델 자체의 한계는 어쩔 수 없다. 결론: **돈이 충분하면 그냥 제일 비싼 모델(Opus)로 미는 게 제일 속 편하다.** 예산이 빠듯할 때만 "모델별 장단점을 파악해서 프로젝트의 각 단계(설계/구현/디테일 다듬기 등)에 적절한 모델을 배분"하는 전략이 의미가 있다.

## 개념 요약

### 시도한 순서와 결과 체감

| 시도 | 체감 결과 |
|---|---|
| No Plan | 나쁘지 않음, 그러나 남들과 비슷한 결과물 |
| Plan + 방향성 질문 | 질문이 기본적이라 크게 안 벗어남 |
| `frontend-design` 스킬 | 조금 더 나음, "한 방에 완성"은 아님 |
| 스킬 + 타 사이트 design.md | 새로 만드는 게 아니라 "가져다 쓰는" 것 |
| 하위 모델(Haiku) | 모델 한계는 그대로 드러남 |

**실무 예시**: 이 표를 순서대로 겪어보면, "도구(스킬/플랜/참고자료)가 좋아질수록 결과가 좋아진다"기보다는 **"내가 원하는 바를 얼마나 구체적으로 알고 지시하느냐"가 결과 품질의 진짜 상한선**이라는 걸 체감하게 된다. 도구는 그 상한선 안에서 도달 속도를 높여줄 뿐이다.

### 핵심 결론: 좋은 결과는 "내 니즈를 아는 것"에서 시작한다

- 경험(다양한 디자인을 직접 본 것)이 쌓이면 → 클로드가 묻기 전에 내가 먼저 구체적으로 지시 가능
- 경험이 부족하면 → 클로드에게 "세부적으로 캐물어 달라"고 먼저 요청해서, 나조차 몰랐던 니즈를 대화로 끌어내야 함

**실무 예시**: 레퍼런스를 많이 본 사람일수록 "이런 느낌으로"가 아니라 "그리드는 12컬럼, 포인트 컬러는 채도를 낮춘 코랄"처럼 구체적으로 요청할 수 있다. 경험이 부족한 단계에서는 오히려 "질문을 많이 받는 것" 자체가 실력을 대신할 수 있는 전략이 된다.

### 모델 선택 전략

| 예산 상황 | 전략 |
|---|---|
| 넉넉함 | 고민 없이 최상위 모델(Opus 등)로 전 과정 진행 |
| 빠듯함 | 모델별 장단점을 파악해서, 프로젝트 단계별(기획/구현/디테일)로 적절한 등급의 모델을 배분 |

**실무 예시**: 예산이 제한된 팀에서는 초안/반복 실험 단계는 저렴한 모델로 빠르게 돌리고, 최종 완성도를 다듬는 단계에서만 상위 모델로 전환하는 "모델 티어링" 전략을 쓰기도 한다.
