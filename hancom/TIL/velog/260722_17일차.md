## 도구 & 환경

### 콘다 가상환경이 여러 개일 때 ModuleNotFoundError 잡는 법

`pip install pytesseract`로 분명히 설치했는데도 실행하면 `ModuleNotFoundError: No module named 'pytesseract'`가 반복되는 상황이 있었다. 원인은 패키지가 없어서가 아니라, **VSCode가 실행하는 파이썬 인터프리터와 패키지가 설치된 환경이 서로 달랐던 것**이다.

```bash
conda env list          # base, nlp, ocrpy, ocrpy38, py39 등 여러 환경 확인
pip show pytesseract    # 어느 환경에 설치돼 있는지 확인
python -c "import sys; print(sys.executable)"   # 지금 실행 중인 인터프리터 확인
```

패키지는 base 환경에만 설치돼 있는데, IDE가 실제로는 ocrpy38 같은 다른 환경으로 파일을 실행하고 있으면 계속 같은 에러가 난다.

**실무 예시**: 콘다/venv를 여러 개 쓰는 프로젝트에서 "분명히 설치했는데 안 된다"는 에러는 십중팔구 패키지 문제가 아니라 인터프리터 불일치다. 에러 메시지만 보지 말고 `sys.executable`과 `pip show`를 나란히 찍어 비교하는 습관이 원인 파악을 훨씬 빠르게 해준다.

## 코딩 학습

### MCP와 공공 API로 AI 할루시네이션 줄이기

MCP(Model Context Protocol)는 AI에게 새 능력을 추가해주는 "USB 같은 것"이다. LLM은 학습 시점까지의 지식만 갖고 있어서 실시간 정보나 모르는 내용은 그럴듯하게 지어내는 할루시네이션(hallucination) 문제가 있는데, MCP로 외부 도구(API, DB 등)를 연결해주면 AI가 실제 데이터를 근거로 답할 수 있게 된다.

대표적인 활용 사례가 공공 CCTV API와 YOLO의 결합이다. 공공데이터 포털에서 실시간 CCTV 스트림 주소를 API로 받아온 뒤, 그 영상을 YOLO 같은 비전 모델에 그대로 넣어 실시간 분석(차량 수 집계, 정체 판단 등)을 수행하는 구조다.

**실무 예시**: "사실 확인은 API/모델이, 설명은 LLM이" 담당하도록 역할을 나누는 게 핵심이다. 실시간 교통량을 묻는 챗봇이라면 LLM에게 바로 답하게 하지 않고, 공공 API로 실제 CCTV 영상을 가져와 YOLO로 집계한 숫자를 근거로 넘긴 뒤 LLM은 그 숫자를 자연어로 설명하는 역할만 맡기는 식이다.

### HuggingFace InferenceClient로 LLM 채팅과 이미지 생성

HuggingFace는 전 세계 AI 연구자가 만든 모델·데이터셋·데모를 무료로 공유하는 플랫폼이다. Models(모델 저장소), Datasets(데이터셋), Spaces(Gradio/Streamlit 데모 호스팅) 세 축으로 구성되고, `InferenceClient`를 쓰면 모델을 로컬에 받지 않고도 API 한 줄로 원격 호출이 가능하다.

```python
from huggingface_hub import InferenceClient

client = InferenceClient(provider="auto")

# LLM 채팅 — OpenAI API와 동일한 인터페이스 구조
completion = client.chat.completions.create(
    model="deepseek-ai/DeepSeek-V3.2:novita",
    messages=[{"role": "user", "content": "질문 내용"}],
)
print(completion.choices[0].message)

# 텍스트 → 이미지 생성 (TTI)
image = client.text_to_image(
    "cute cat sitting on a rooftop, sunset, anime style, high quality",
    model="black-forest-labs/FLUX.1-dev",
)
image.save("result.jpg")
```

`chat.completions.create()` 구조가 OpenAI API와 거의 똑같이 생긴 건 우연이 아니라, 업계 표준처럼 굳어진 인터페이스를 그대로 따른 것이다. 이미지 생성 프롬프트는 영어로, 스타일 키워드(`anime style`, `photorealistic`)와 품질 키워드(`high quality`, `4K`)를 같이 넣어야 결과물 퀄리티가 확실히 올라간다.

아바타 생성 앱처럼 사용자 입력을 받는 서비스라면, 입력을 그대로 모델에 넣지 않고 f-string으로 스타일 문구를 감싸서 보강한다.

```python
cute_prompt = (
    f"Cute adorable character illustration, "
    f"lovely kawaii style: {user_description}. "
    f"Bright colors, friendly expression, digital art, high quality"
)
image = client.text_to_image(cute_prompt, model="black-forest-labs/FLUX.1-dev")

# 생성 기록은 CSV로 누적 저장
new_record = {"생성일시": timestamp, "이름": user_name, "설명": user_description, "이미지파일": image_path}
df = pd.concat([df, pd.DataFrame([new_record])])
df.to_csv("avatar_records/avatar_records.csv", index=False)
```

**실무 예시**: 사용자에게 프롬프트를 직접 쓰게 하지 않고 서비스 쪽에서 스타일 문구를 강제로 붙여 결과물 톤을 일정하게 유지하는 방식은, 캐릭터 생성 이벤트나 커머스 앱의 "내 캐릭터 만들기" 기능에 그대로 쓸 수 있는 패턴이다.

### Transformers로 문장 유사도·감정분석·요약+번역

`pipeline()` 함수는 태스크 이름만 바꿔주면 알맞은 사전학습 모델과 후처리를 자동으로 연결해주는 HuggingFace의 고수준 API다.

**문장 유사도**는 `SentenceTransformer`로 문장을 384차원 벡터(임베딩)로 바꾼 뒤, 두 벡터의 코사인 유사도로 의미적 유사성을 수치화한다.

```python
from sentence_transformers import SentenceTransformer, util

model = SentenceTransformer("all-MiniLM-L6-v2")
emb1 = model.encode("He is reading a book in the library", convert_to_tensor=True)
emb2 = model.encode("He is at the library reading a book", convert_to_tensor=True)
cos_sim = util.pytorch_cos_sim(emb1, emb2)   # ≈ 0.94, 표현이 달라도 의미가 비슷하면 높게 나옴
```

**감정분석**은 `label`(POSITIVE/NEGATIVE)과 `score`(확신도)를 반환한다. `score`는 모델의 raw 출력값을 Softmax로 0~1 확률값으로 변환한 것이다.

```python
from transformers import pipeline
classifier = pipeline("sentiment-analysis")
results = classifier("I'm feeling really great today")
# results[0]['label'] → POSITIVE, results[0]['score'] → 0.99 근처
```

**요약+번역**은 T5(`t5-small`)가 영어 전용이라는 한계를, 구글 번역으로 보완하는 2단계 파이프라인으로 처리한다.

```python
from deep_translator import GoogleTranslator

summarizer = pipeline("summarization", model="t5-small")
summary = summarizer(text, min_length=20, max_length=60)
kr_text = GoogleTranslator(source='en', target='ko').translate(summary[0]['summary_text'])
# 원문(영) → T5 요약 → 요약(영) → GoogleTranslator → 요약(한)
```

T5 같은 인코더-디코더 모델은 `"summarize: "`, `"translate English to Korean: "`처럼 태스크 접두사(Task Prefix)로 "이번엔 무슨 작업인지" 지시받는데, `pipeline`을 쓰면 이 접두사가 자동으로 붙는다.

**실무 예시**: 코사인 유사도는 고객 문의를 FAQ와 매칭하는 챗봇 검색이나 중복 리뷰 탐지에, 감정분석은 SNS·리뷰 모니터링으로 브랜드 평판을 자동 집계할 때, 요약+번역 파이프라인은 해외 뉴스를 한국어로 자동 브리핑하는 서비스에 그대로 쓰인다.

### Tesseract OCR — 빈 결과와 깨진 결과의 진짜 원인

`pytesseract.image_to_string()`으로 이미지에서 글자를 뽑는 실습에서, 에러 없이 결과만 이상하게 나오는 두 가지 케이스를 직접 겪었다.

**① 결과가 아예 빈 문자열인 경우**: pytesseract는 Tesseract 내부 경고성 오류를 예외로 던지지 않고 조용히 넘어간다. 원본 tesseract.exe를 터미널에서 직접 실행해보면 원인이 드러난다.

```
tesseract 이미지파일 stdout -l eng
# Error in boxClipToRectangle: box outside rectangle
# Error in pixScanForForeground: invalid box
```

사진 속에 이미지 경계에 걸쳐 지나가는 가는 선(벽의 균열 등)이 있으면, 자동 레이아웃 분석이 이걸 텍스트 영역으로 착각해 경계를 벗어나는 바운딩 박스를 계산하며 내부적으로 깨진다.

**② 글자가 뻔히 보이는데 완전히 의미 없는 문자열이 나오는 경우**: `--psm 6`으로 위 크래시는 피했지만, 이미지가 4000×6000처럼 초고해상도라 글자 획 높이가 수백 px에 달하면 Tesseract의 문자 분류기가 기대하는 정상 범위(대략 20~40px)를 벗어나 벽 질감 노이즈까지 전부 글자로 오인식한다. 해결은 그레이스케일 → 이진화 → 리사이즈다.

```python
gray = image.convert("L")
binarized = gray.point(lambda p: 255 if p > 150 else 0)
ratio = 800 / binarized.width
resized = binarized.resize((800, int(binarized.height * ratio)))
results = pytesseract.image_to_string(resized, lang='eng', config='--psm 6')
```

반대로 342×584처럼 **너무 작은** 스크린샷에서는 한글은 그럭저럭 읽혀도 그 안에 섞인 영어 단어(`follow-up`, `feedback` 등)만 유독 깨진다. 한글은 초성·중성·종성이 결합된 복잡한 형태라 저해상도에서도 버티지만, 단순한 획으로 이루어진 알파벳은 몇 px에서 뭉개지기 쉽기 때문이다. 3배 업스케일로 해결된다.

```python
w, h = image.size
resized = image.resize((w * 3, h * 3), Image.LANCZOS)
results = pytesseract.image_to_string(resized, lang='eng+kor')
```

`Image.LANCZOS`는 없던 정보를 만들어내는 AI 초해상도가 아니라, 기존 픽셀을 부드럽게 재배치하는 보간 방식이다. 핵심은 "사람 눈에 잘 보이는가"가 아니라 **Tesseract 내부 정규화 파이프라인이 기대하는 글자 크기(20~40px)에 맞추는 것**이라, 원본이 너무 크면 축소, 너무 작으면 확대가 정답이다.

Tesseract 같은 전통 OCR은 "픽셀 뭉치가 어떤 글자 모양과 비슷한가"를 사전·n-gram으로 얕게 보정하는 시각 패턴 매칭이 전부라, 문맥이 말이 되는지는 이해하지 못한다. Claude·GPT 같은 Vision-Language 모델은 시각 인식과 언어 이해를 통합해서 문맥으로 오류를 자가검증할 수 있지만, 이미지 1장당 비용·속도 부담이 훨씬 크다. 그래서 실무에서는 Tesseract로 대량을 빠르게 1차 처리하고, 확신도가 낮은 부분만 VLM으로 재검증하는 하이브리드 구조를 쓴다.

**실무 예시**: 영수증·명함 스캔 앱은 촬영 각도·해상도가 제각각이라 "원본 상태를 먼저 판단해서 그에 맞는 전처리(축소 or 확대)를 자동 적용"하는 단계가 항상 들어간다. 대량 문서 디지털화에는 비용 때문에 Tesseract를, 정확도가 중요한 소량·검수 단계에는 VLM을 쓰는 하이브리드 구조가 일반적이다.

## 핵심 정리
- "분명히 설치했는데 안 된다"는 에러는 패키지 문제가 아니라 콘다 인터프리터 불일치인 경우가 많다 — `sys.executable`로 확인
- MCP는 AI에게 도구를 붙여주는 규격이고, 공공 API로 실시간 사실 데이터를 근거로 줘야 할루시네이션을 줄일 수 있다
- HuggingFace `InferenceClient`는 OpenAI API와 유사한 인터페이스로 LLM 채팅·이미지 생성을 API 한 줄로 호출한다
- Transformers `pipeline()`은 태스크명만 바꾸면 문장유사도·감정분석·요약 등을 통일된 방식으로 처리하고, 영어 전용 모델은 번역 후처리로 보완한다
- Tesseract OCR은 에러 없이 결과만 이상하게 나올 때가 많다 — 빈 결과는 Leptonica 내부 크래시, 깨진 결과는 글자 크기가 정규화 범위(20~40px)를 벗어난 게 원인이며, 이미지 해상도에 따라 축소/확대로 대응해야 한다
