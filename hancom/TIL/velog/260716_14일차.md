## 도구 & 환경

### 가상환경, 패키지는 설치했는데 왜 안 될까

`pyfiglet`, `termcolor`를 쓰는 스크립트를 실행했더니 `ModuleNotFoundError`가 났다. 그런데 분명 가상환경에 설치해둔 패키지였다. 확인해보니 원인은 단순했다 — **패키지 자체는 정상 설치돼 있었지만, 지금 실행 중인 파이썬 인터프리터가 그 가상환경이 아니었다.**

```bash
conda env list
# base                     (활성 상태 *)
# py39                     ← 패키지는 여기 설치돼 있음

conda run -n py39 python -c "import termcolor, pyfiglet"
# OK
```

`base` 환경으로 실행하면 `py39`에 설치된 패키지가 안 보이는 게 당연하다. VS Code라면 우측 하단(또는 `Ctrl+Shift+P` → "Python: Select Interpreter")에서 인터프리터를 바꿔주거나, 터미널에서 `conda activate py39` 후 실행하면 해결된다.

이 문제는 실무에서 "내 컴퓨터에선 되는데 팀원 컴퓨터/서버에선 안 된다"는 이슈의 단골 원인이기도 하다. 배포 스크립트나 CI 환경에서도 지금 어떤 인터프리터/가상환경이 활성화돼 있는지 명시적으로 확인하는 습관이 필요하다.

## 코딩 학습

### zip, 두 리스트를 인덱스 기준으로 묶기

`zip()`은 값의 내용을 비교해서 짝짓는 게 아니라, **각 리스트의 인덱스 순서대로 그대로 묶어준다.** `names[0]`↔`scores[0]`, `names[1]`↔`scores[1]`처럼 위치 기준으로만 매칭되기 때문에, 두 리스트가 실제로 같은 대상을 순서대로 가리키고 있어야 의미가 맞다.

```python
names = ["뽀삐", "초코", "쿠키"]
scores = [95, 88]

for name, score in zip(names, scores):
    print(f"{name}: {score}점")
# 뽀삐: 95점
# 초코: 88점
# → "쿠키"는 짝지을 점수가 없어서 에러 없이 조용히 버려짐
```

길이가 다르면 더 짧은 쪽 기준으로 자동으로 멈추고, 남는 원소는 경고 없이 버려진다.

| 상황 | 결과 |
|---|---|
| 두 리스트 길이 같음 | 끝까지 정상적으로 짝지어짐 |
| 길이 다름 | 짧은 쪽 길이에 맞춰 멈춤 (에러 없음) |
| 순서가 어긋난 리스트 | 값이 아니라 위치로 묶이므로 엉뚱한 값끼리 짝지어짐 |

**실무 예시**: DB에서 뽑은 "컬럼명 리스트"와 "값 리스트"를 한 줄씩 짝지어 딕셔너리로 만들 때(`dict(zip(columns, row))`) 자주 쓰인다. 이때 두 리스트 순서가 어긋나면 엉뚱한 값이 엉뚱한 컬럼에 들어가므로, 순서 보장이 중요하다.

### return과 print는 다른 동작이다

함수가 `return`으로 값을 돌려주는데도, 그 함수를 호출만 하고 결과를 안 받으면 콘솔에 아무것도 안 찍힌다. 함수 자체엔 문제가 없다 — `return`은 값을 호출한 곳으로 "전달"만 할 뿐, 화면에 출력해주는 게 아니기 때문이다.

```python
def highlight(text: str, color: str) -> str:
    return colored(text, color)

# 이렇게만 호출하면 아무것도 안 보임
highlight("good", "yellow")

# 반환값을 받아서 직접 출력해야 함
results = highlight("good", "yellow")
print(results)
```

**실무 예시**: 함수가 값을 반환하게 만들어야 그 결과를 로그로 남기거나, 다른 함수에 다시 넘기거나, 테스트로 검증할 수 있다. 함수 안에서 바로 `print`만 하면 그 값을 다른 곳에서 재사용할 방법이 없어진다.

### 클래스 — 속성과 메서드로 데이터와 동작 묶기

클래스는 "붕어빵 틀"에 비유하면 이해가 쉽다. `class World`는 틀을 만드는 것이고, `__init__`은 틀로 찍을 때 자동으로 이름표(속성)를 붙이는 과정, `self`는 "이 붕어빵 자기 자신"을 가리킨다. `World("Korea")`는 틀로 붕어빵 하나를 찍어 이름표에 "Korea"를 적은 것이고, `asia.hello()`는 그 붕어빵한테 재주(메서드)를 시키는 것이다.

```python
class World:
    def __init__(self, name, population):
        self.name = name
        self.population = population
    def hello(self):
        print(f"Hello,{self.name}!!")
    def info(self):
        print(f"{self.name}의 인구는 {self.population}명입니다.")

asia = World("Korea", 51000000)
asia.hello()   # Hello,Korea!!
asia.info()    # Korea의 인구는 51000000명입니다.
```

`population` 속성과 `info()` 메서드는 오늘 새로 추가해본 부분이다. 관련 데이터(`name`, `population`)와 그 데이터를 다루는 동작(`hello`, `info`)을 하나의 틀 안에 묶어두면, 객체를 몇 개를 만들든 같은 구조를 재사용할 수 있다.

| 용어 | 역할 |
|---|---|
| `class` | 설계도(틀) |
| `__init__` | 객체 생성 시 자동 실행되는 초기화 |
| `self` | 객체 자기 자신을 가리키는 참조 |
| 속성 | 클래스 안의 변수 (`self.name`) |
| 메서드 | 클래스 안의 함수 (`hello()`, `info()`) |

**실무 예시**: 사용자, 상품, 주문처럼 "이름표(데이터)"와 "행동(로직)"이 함께 필요한 대상을 다룰 때 클래스를 쓴다. 예를 들어 `User` 클래스에 `name`, `email` 같은 속성과 `send_welcome_email()` 같은 메서드를 함께 묶어두면, 객체 하나로 데이터와 관련 동작을 한 번에 관리할 수 있다.

### 람다 — 함수를 한 줄로 압축하기

`def`로 만든 함수 중 본문이 "여러 문장"처럼 보여도, 결국 중간 변수 없이 한 줄 표현식으로 압축할 수 있는 경우가 많다. `lambda`는 이런 함수를 `이름 = lambda 매개변수: 표현식` 형태로 축약해준다.

```python
# before
def meters_to_feet(meters):
    feet = meters * 3.28084
    print(feet)

# after — 중간 변수 없이 한 줄로
meters_to_feet = lambda meters: print(meters * 3.28084)
```

중간 변수가 2개 있던 함수(pyfiglet + termcolor 조합)도 함수 호출을 안쪽에 겹쳐 넣으면 압축된다:

```python
good_sentence = lambda sentence: print(colored(pyfiglet.figlet_format(sentence), "blue", "on_green"))
```

다만 두 가지 함정을 실습하면서 확인했다.

**1) `print()`는 색을 입혀주지 않는다.** `print(문자열, "green", "on_yellow")`처럼 색상 인자를 그냥 `print()`에 넘기면, `print()`는 받은 인자들을 공백으로 이어서 그대로 출력할 뿐이라 `"green"`, `"on_yellow"`가 글자 그대로 찍힌다. 색을 입히려면 `colored()`로 먼저 색상 코드를 입힌 뒤, 그 결과 하나만 `print()`에 넘겨야 한다.

**2) lambda 안에서 `print()`를 또 감싸면 `None`이 따라온다.** 본문이 `print(...)`인 lambda는 호출 시 화면에 이미 출력하고, 반환값은 `print()`의 반환값인 `None`이 된다. 이 lambda를 바깥에서 또 `print(st("hello"))`처럼 감싸면, 그 `None`을 다시 출력하게 된다.

```python
# 함정: 출력형 lambda를 바깥에서 또 print로 감쌈
st = lambda st: print(pyfiglet.figlet_format(st))
print(st("hello"))   # 아스키아트 밑에 None이 따라 나옴

# 해결: 이미 안에서 출력하니 그냥 호출만
st("hello")

# 반환형 lambda라면 바깥에서 print로 감싸는 게 맞음
decorate_text = lambda text: pyfiglet.figlet_format(text)
print(decorate_text("LAMBDA"))
```

즉 lambda는 "직접 출력하는 형태"와 "값만 반환하는 형태" 두 가지로 나뉘고, 용도에 따라 구분해서 써야 한다.

| lambda 형태 | 반환값 | 바깥에서 print 필요? |
|---|---|---|
| 본문이 `print(...)` | 항상 `None` | 필요 없음 (이미 출력함) |
| 본문이 값 표현식 | 계산 결과 값 | 필요함 |

**실무 예시**: `sorted(data, key=lambda x: x["age"])`처럼 정렬 기준을 짧게 넘기거나, `map()`/`filter()`와 함께 한 줄짜리 변환 로직을 즉석에서 쓸 때 lambda가 흔히 쓰인다. 이때는 항상 "값을 반환하는" lambda여야 한다 — `key=`나 `map()`은 다음 단계로 넘길 값이 필요하지, 화면 출력이 필요한 게 아니기 때문이다.

## 핵심 정리
- `zip()`은 값이 아니라 인덱스 위치로 짝짓고, 길이가 다르면 짧은 쪽 기준으로 조용히 멈춘다
- `return`은 값 전달, `print`는 화면 출력 — 별개의 동작이라 반환값을 보려면 직접 `print()`로 감싸야 한다
- 패키지가 설치돼 있어도 실행 중인 인터프리터(가상환경)가 다르면 못 찾는다 — 항상 활성 환경을 확인하는 습관이 필요하다
- 클래스는 속성(데이터)과 메서드(동작)를 하나의 틀로 묶어, 객체를 몇 개 만들든 같은 구조를 재사용하게 해준다
- lambda는 "직접 출력하는 형태"와 "값만 반환하는 형태"를 구분해서 써야 하며, `key=`처럼 값이 필요한 자리에는 반드시 반환형을 써야 한다
