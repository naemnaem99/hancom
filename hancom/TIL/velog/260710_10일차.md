## 도구 & 환경

### npm run dev는 결국 Node.js 서버다

`npm run dev`로 뜨는 로컬 서버는 결국 Node.js 프로세스다. `package.json`의 `scripts.dev`에 적힌 명령(Vite 프로젝트라면 보통 `vite`)을 실행하는 것이고, 그 `vite`라는 프로그램 자체가 JS로 작성되어 Node.js 위에서 동작한다. 터미널에서 실행하는 순간 Node.js 프로세스 하나가 백그라운드에 떠서 `Ctrl+C`로 끄기 전까지 계속 요청을 받아 응답한다.

| 층 | 역할 |
|---|---|
| npm | `package.json`을 보고 `dev`가 뭘 실행할지 찾아주는 실행기 |
| vite | 실제로 개발 서버를 띄우는 프로그램 (JS로 작성됨) |
| Node.js | vite라는 JS 코드를 실제로 돌려주는 런타임 |

VSCode Live Server("Go Live")와의 가장 큰 차이는 **JS를 변환해주는지 여부**다.

| | Live Server | Vite dev server |
|---|---|---|
| 대상 | 순수 HTML/CSS/JS | React(JSX) 등 빌드가 필요한 프로젝트 |
| JSX 처리 | 못함 (에러) | 실시간 변환(트랜스파일) |
| 저장 시 갱신 방식 | 전체 새로고침 | HMR (상태 유지하며 부분 갱신) |
| npm 패키지 사용 | 불가 | 가능 |

Live Server는 파일을 있는 그대로 서빙하는 정적 파일 서버라 JSX나 `import`/`export` 같은 모듈 문법을 이해하지 못한다. 반면 Vite는 JSX를 즉석에서 변환하고, 저장할 때도 전체 새로고침이 아니라 바뀐 컴포넌트만 갈아끼우는 HMR(Hot Module Replacement)로 `useState` 값 같은 상태를 유지한 채 화면만 갱신한다.

**실무 예시**: React 프로젝트는 Live Server로 절대 못 띄우고, 반대로 순수 HTML 실습 폴더는 Vite 없이 Live Server만으로 충분하다. 배포할 때는 `npm run build`로 순수 정적 파일만 뽑아내므로, 그 결과물은 Node.js 없이 Nginx나 GitHub Pages 같은 정적 호스팅에서도 그대로 돌아간다.

## 코딩 학습

### 언어들은 "어디서 실행되는가"로 나뉜다

HTML/CSS/JS는 전부 **브라우저**라는 같은 장소에서 실행되는 삼총사라 자연스럽게 하나로 묶여서 이해된다. 반면 React·Node.js·Java·C는 "다음 단계"가 아니라 **서로 다른 위치에서 실행되는 완전히 다른 갈래**다. "이 코드가 실제로 어디서 실행되는가"를 기준으로 나누면 훨씬 명확해진다.

```
[ 브라우저 (프론트엔드) ]           [ 서버 (백엔드) ]
   HTML / CSS / JS                    Node.js, Java, C...
   React (여기 포함)                   Express, Spring...
        └────────── HTTP로 통신 ──────────┘
```

React는 새로운 언어가 아니라 JS로 화면을 더 편하게 짜도록 도와주는 **라이브러리**다. `.jsx` 파일도 Vite가 빌드하면 결국 평범한 HTML/CSS/JS로 변환돼서 브라우저에 전달되므로, 실행 위치는 순수 JS와 똑같이 브라우저다.

| 위치 | 기술 | 정체 |
|---|---|---|
| 브라우저 | HTML/CSS/JS | 화면의 뼈대·디자인·동작 |
| 브라우저 | React | JS로 화면을 컴포넌트 단위로 짜는 라이브러리 |
| 브라우저/서버 공통 | TypeScript | JS/Node 위에 타입 검사를 얹는 컴파일 도구 |
| 서버 | Node.js + Express | JS로 만드는 웹 서버 |
| 서버 | Java | JVM 위에서 도는 별도 언어, 대기업 백엔드/안드로이드 |
| 시스템 | C | 운영체제·임베디드 등 하드웨어에 가까운 프로그램 |
| 기기 자체(iOS) | Swift | 앱스토어에 올라가는 네이티브 앱 |
| 데이터베이스 내부 | PL/pgSQL | DB 안에서 도는 로직(트리거·함수) |
| 컴퓨터/인프라 | Shell | 명령어 자동화, 배포, 서버 관리 |

TypeScript는 새로운 실행 위치가 아니라 JS 위에 "타입 검사"라는 한 겹을 더 씌운 것이다. 반드시 컴파일러(`tsc`)를 거쳐 평범한 JS로 변환된 뒤에야 실행된다.

**실무 예시**: 쇼핑몰 서비스 하나를 예로 들면 — 사용자가 보는 화면은 React(브라우저), 주문 요청을 받는 서버는 Node.js/Express, 재고 차감처럼 정합성이 중요한 로직은 DB 안의 PL/pgSQL 함수, 서버 배포 자동화는 Shell 스크립트, iOS 전용 앱도 함께 낸다면 Swift로 별도 제작 — 이렇게 한 서비스 안에 여러 언어가 각자 자기 위치를 맡아 동시에 존재한다.

### React: 조건부 렌더링

React에서 조건에 따라 다른 화면을 보여주는 방식은 경우의 수에 따라 다르게 고른다.

| 방식 | 문법 | 쓰는 경우 |
|---|---|---|
| `&&` | `{cond && <p/>}` | 보이거나/안 보이거나 2択 |
| 삼항 연산자 | `{cond ? A : B}` | 둘 중 하나를 항상 보여줌 |
| 객체 매핑 | `{ {a:'x', b:'y'}[type] }` | 경우의 수가 3개 이상일 때 |

**실무 예시**: 로그인 여부에 따라 "로그인"/"로그아웃" 버튼 중 하나만 보여줄 땐 `&&`나 삼항, 주문 상태("결제대기"/"배송중"/"배송완료"/"취소")처럼 상태가 여러 개고 상태별 색상까지 정해져 있으면 객체 매핑으로 관리하는 게 조건문이 계속 늘어나는 걸 막아준다.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### React: controlled input

input의 현재 값을 브라우저(DOM)가 아니라 React state가 관리하는 방식을 **controlled input**이라 한다.

```jsx
const [name, setName] = useState('')
<input value={name} onChange={(e) => setName(e.target.value)} />
```

**실무 예시**: 로그인 폼처럼 입력 도중 "8자 이상인지", "이메일 형식이 맞는지" 실시간으로 검증하거나, 검색창에서 타이핑할 때마다 자동완성 목록을 갱신하려면 입력값을 state로 갖고 있어야 한다.

### React: 리스트 렌더링 — map, key, 반복 트릭

```jsx
{users.map((u) => (
    <li key={u.id}>{u.name}</li>
))}
```

`key`는 React가 리스트의 각 항목을 구분해서 추적하기 위한 고유 식별자다. 없으면 리스트가 바뀔 때(추가/삭제/순서변경) 어떤 항목이 바뀌었는지 몰라서 엉뚱한 항목이 갱신될 수 있다. 배열 인덱스보다는 `id`처럼 데이터 고유값을 쓰는 게 안전하다. `.map()` 콜백의 매개변수 이름(`u`, `k` 등)은 임의로 붙인 이름이라 뭘 쓰든 상관없지만, `id`/`name` 같은 속성명은 API 응답에 고정된 이름이라 그대로 써야 한다.

데이터가 없이 그냥 "정해진 횟수만큼" 반복하고 싶을 땐 `[...Array(n)]` 트릭을 쓴다.

```jsx
{[...Array(5)].map((_, i) => (
    <span key={i}>{i < score ? "⭐" : "💥"}</span>
))}
```

`Array(5)`만 쓰면 길이만 5인 빈 슬롯이라 `.map()`이 순회하지 않는다. 스프레드(`[...]`)로 실제 값이 채워진 배열로 바꿔야 정상적으로 5번 돈다.

**주의할 점**: `.map()` 콜백 매개변수 이름을 바깥 컴포넌트/변수 이름과 겹치게 짓지 말 것.

```jsx
const Tag = ({tags}) => (
    <div>
        {tags.map((Tag) => (   // 컴포넌트 자신과 이름이 같음 (섀도잉)
            <span key={Tag}>{"#" + Tag}</span>
        ))}
    </div>
)
```

콜백 매개변수가 하필 바깥 컴포넌트 이름과 똑같으면 그 스코프 안에서 원래 이름을 가려버리는(섀도잉) 코드가 된다. 지금은 우연히 동작해도, 나중에 콜백 안에서 진짜 `Tag` 컴포넌트를 참조해야 할 때 찾기 힘든 버그가 된다.

**실무 예시**: 별점처럼 서버 데이터가 아니라 정해진 횟수만큼 반복 렌더링할 때 `[...Array(n)]`을 쓰고(페이지네이션 버튼, 스켈레톤 카드 등), 콜백 매개변수 섀도잉은 리뷰에서 자주 지적되는 실수라 ESLint `no-shadow` 규칙을 켜두는 팀도 많다.

### React: state 갱신 — setState 함수형 업데이트

```jsx
// 현재 state 값을 직접 참조
<button onClick={() => setCount(count + 1)}>{count}</button>

// 함수형 업데이트
<button onClick={() => setCount(c => c + 1)}>{count}</button>
```

`setCount(count + 1)`은 그 순간 렌더링에 캡처된 값을 기준으로 계산해서, 한 핸들러 안에서 여러 번 연달아 호출하면 마지막 호출만 반영되는 함정이 있다. `setCount(c => c + 1)`처럼 함수형으로 넘기면 React가 항상 최신 값을 넣어 호출해주므로 여러 번 호출해도 정확히 누적된다.

**실무 예시**: 더블클릭 방지 없이 빠르게 연타되는 버튼(장바구니 담기, 좋아요)이나 비동기 콜백 안에서 이전 state를 기준으로 갱신해야 하는 경우 함수형 업데이트를 기본값으로 쓰는 게 안전하다.

### React: MUI 컴포넌트 조합

MUI 컴포넌트는 다음 조합으로 빠르게 통일된 UI를 만든다.

| 컴포넌트 | 역할 |
|---|---|
| `AppBar` + `Toolbar` | 상단 바 + 내부 레이아웃/여백 |
| `Tabs` + `Tab` | 탭 전환 UI, `value`/`onChange`로 controlled 사용 |
| `Card` + `CardContent` | 그림자 있는 콘텐츠 박스 |
| `TextField` | 입력창 (label, variant 등 내장) |
| `sx` prop | MUI 전용 인라인 스타일. 숫자는 8px 배수, 테마 토큰/가상선택자 사용 가능 |

탭마다 다른 콘텐츠를 보여주려면 선택 상태(`value`)를 자식이 아니라 **부모로 끌어올려야(state lifting)** 한다.

```jsx
// App.jsx
const [tab, setTab] = useState(0)
<Menu value={tab} onChange={(_, v) => setTab(v)} />
<Content tab={tab} />
```

`sx`는 **MUI 컴포넌트 전용** prop이라 일반 `<button>`/`<div>`엔 안 먹힌다. 순수 HTML 요소는 표준 `style` 속성을 그대로 쓴다.

```jsx
<button style={{ border: "2px solid #5e0f0f" }}>담기</button>   {/* 일반 태그 */}
<Button sx={{ mt: 2 }}>제출</Button>                             {/* MUI 컴포넌트 */}
```

컴포넌트는 실습 폴더 번호와 상관없이 어디서든 import해서 재사용할 수 있다 — `Header`가 `Counter`를 그대로 가져다 `AppBar` 안에 끼워 넣는 식이다.

**실무 예시**: 관리자 대시보드나 쇼핑몰 상단 네비게이션을 만들 때 `AppBar`+`Toolbar`로 뼈대를 잡고, 카테고리 전환은 `Tabs`, 상품 카드 목록은 `Card`, 검색/로그인 입력창은 `TextField`로 조립하면 디자인 시스템 없이도 빠르게 통일된 UI를 낼 수 있다.

### React: flex 균등분배

`flexGrow`만 주면 `flex-basis`는 기본값 `auto`(콘텐츠 크기)로 남아서, 빈 박스와 내용이 있는 박스가 시작 크기부터 다르면 정확히 균등 분배되지 않는다.

| 속성 | flex-basis | 결과 |
|---|---|---|
| `flexGrow: 1` | `auto` (콘텐츠 크기 유지) | 시작 크기가 다르면 비율이 어긋남 |
| `flex: 1` | `0%` | 시작 크기 무시하고 완전 균등 분배 |

```jsx
<Box sx={{ display: 'flex' }}>
    <Box sx={{ flex: 1 }}>...</Box>
    <Typography sx={{ flex: 1, textAlign: 'center' }}>제목</Typography>
    <Box sx={{ flex: 1 }} />
</Box>
```

**실무 예시**: 헤더에 "로고 | 메뉴 | 검색창/로그인 버튼"처럼 좌우 요소 크기가 다른데도 가운데를 정확히 맞추고 싶을 때 3분할 `flex: 1` 패턴을 쓴다.

### React: useEffect 의존성 배열과 cleanup

`useEffect`의 의존성 배열은 세 가지 경우로 실행 시점이 달라진다.

| 배열 | 실행 시점 |
|---|---|
| 없음 | 렌더링될 때마다 매번 |
| `[]` | 마운트 시 딱 1번 |
| `[값]` | 마운트 시 1번 + 그 값이 바뀔 때마다 |

```jsx
useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000)
    return () => clearInterval(id)  // cleanup: 재실행/unmount 직전 호출
}, [])
```

`useEffect` 안에서 `return`한 함수는 cleanup으로 등록되어, 재실행되거나 컴포넌트가 사라질 때(unmount) 자동 호출된다. 타이머를 정리하지 않으면 계속 쌓여서 중복 실행되는 메모리 누수가 생긴다.

**헷갈리기 쉬운 포인트**: "deps 배열이 아예 없는" 경우(위 표의 "없음")는 안에서 state를 바꾸면 정말 무한루프가 되지만, **`[]`(빈 배열)는 안에서 state를 바꿔도 무한루프가 아니다.**

```jsx
useEffect(() => {
    setCount(c => c + 1)
    console.log("실행")
}, [])
```

빈 배열은 "마운트 시 딱 1번"만 실행되는 조건이라, 안에서 state를 바꿔도 이펙트 자체가 다시 실행되지 않는다 — 딱 1번 추가 렌더링으로 끝난다. 즉 "이펙트 안에서 state를 바꾸는가"가 아니라 **"의존성 배열이 무엇인가"가 재실행 여부를 결정한다.**

**실무 예시**: `[]`는 페이지 진입 시 상품 목록을 한 번만 불러올 때, `[keyword]`처럼 특정 값 의존은 검색어가 바뀔 때마다 목록을 다시 불러오는 검색 기능에 쓴다. cleanup은 실시간 채팅방의 소켓 연결을 컴포넌트를 벗어나면 반드시 끊어줘야 하는 경우에 필수다.

### fetch로 데이터 불러오기

```jsx
const [data, setData] = useState([])
const [isLoading, setIsLoading] = useState(true)

useEffect(() => {
    fetch(URL)
        .then((res) => res.json())
        .then((json) => {
            setData(json)
            setIsLoading(false)
        })
        .catch((error) => {
            console.error(error)
            setIsLoading(false)
        })
}, [])
```

falsy 값(`0`, `""`)이 유효한 데이터일 수 있는 조건문은 주의가 필요하다. `temp ? A : B`처럼 쓰면 기온이 0℃일 때도 `false` 취급되어 "불러올 수 없음"으로 오판된다. `temp !== null ? temp + '℃' : '불러올 수 없음'`처럼 `null` 여부로 명시적으로 검사해야 한다.

`<li>` 안에 `<br/>`를 넣어도 항목 사이 여백은 안 생긴다 — `<br/>`는 같은 요소 안에서 줄바꿈만 만들 뿐, 형제 요소 사이 간격에는 영향을 안 준다. 리스트 항목 간 여백은 부모 `<ul>`에 `gap`을 주는 방식으로 해결해야 한다.

**실무 예시**: 상품 상세 페이지 진입 시 서버에서 재고/가격을 불러와 `isLoading` 동안 스켈레톤 UI를 보여주다가 데이터가 오면 교체하는 패턴이 거의 표준이다. `.catch()`는 네트워크 오류나 서버 다운 시 사용자에게 안내를 보여주는 데 쓴다.

## 핵심 정리
- `npm run dev`는 결국 Node.js 프로세스이며, Live Server와의 핵심 차이는 JSX 변환 여부와 HMR(상태 유지 갱신) 지원 여부
- 언어는 "다음 단계"가 아니라 "실행 위치"로 나눠서 이해해야 함 — 브라우저(HTML/CSS/JS/React), 서버(Node.js/Java), 시스템(C), 기기(Swift), DB(PL/pgSQL), 인프라(Shell)
- React 리스트 렌더링은 `key`가 필수이며, `[...Array(n)]`로 데이터 없이 반복하거나 `setCount(c=>c+1)` 함수형 업데이트를 쓰는 패턴이 자주 헷갈림
- MUI는 `sx` 전용, 일반 태그는 `style` — 컴포넌트 조합과 `flex: 1` 균등분배로 레이아웃을 짠다
- `useEffect`는 의존성 배열([]/[값]/없음)로 실행 시점을 제어하며, `[]`은 안에서 state를 바꿔도 무한루프가 아니라 1회성 재렌더링일 뿐
