## 코딩 학습

### react-router-dom으로 SPA 라우팅하기

my-app2는 `/`, `/whisky`, `/gin`, `/rum` 네 개의 주소를 각각 `Home`, `Whisky`, `Gin`, `Rum` 컴포넌트에 매핑했다. 실제로는 `index.html` 파일 하나뿐이고, 이 네 경로는 서버 파일이 아니라 **브라우저 안에서만 관리되는 가짜 경로**다. `react-router-dom`이 현재 주소를 감시하다가 그에 맞는 컴포넌트로 화면 일부만 바꿔치기 해준다.

```bash
npm install react-router-dom
```

```jsx
// App.jsx
import { BrowserRouter, Routes, Route } from 'react-router-dom'
import Home from './pages/Home.jsx'
import Whisky from './pages/Whisky.jsx'
import Gin from './pages/Gin.jsx'
import Rum from './pages/Rum.jsx'

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/whisky" element={<Whisky />} />
        <Route path="/gin" element={<Gin />} />
        <Route path="/rum" element={<Rum />} />
      </Routes>
    </BrowserRouter>
  )
}
```

| 개념 | 설명 |
|---|---|
| `BrowserRouter` | 앱 전체를 감싸서 라우팅 기능을 활성화하는 컨테이너 |
| `Routes` / `Route` | 경로별로 어떤 컴포넌트를 보여줄지 매핑하는 테이블 역할 |
| `Link` | `<a>`와 달리 페이지 전체를 새로고침하지 않고 주소만 바꿔 컴포넌트를 교체 |

이 앱을 만들면서 실제로 겪은 실수 두 가지가 있다. 첫째, 패키지명을 `react-route-dom`으로 잘못 입력해서 설치 자체가 404로 실패했다 — 올바른 이름은 `router`가 들어간 `react-router-dom`이다. 둘째, `Link`를 소문자 `<link>`로 썼더니 HTML의 스타일시트 연결 태그로 인식돼서 아무 일도 일어나지 않았다 — 반드시 대문자 `Link`로 써야 한다. 그리고 이 프로젝트는 Vite의 변환 과정을 거쳐야만 실행되므로, `index.html`을 Live Server로 직접 열면 JSX와 패키지 이름만 적힌 import를 브라우저가 해석하지 못해 에러가 난다.

### 컴포넌트 구조 설계: pages / components / data / utils

라우팅 개념만 배우고 끝낸 게 아니라, 그 위에 실제로 **칵테일 레시피 앱(`my-app2`)**을 만들면서 폴더 구조를 역할별로 나누는 관례를 익혔다.

```
src/
├── App.jsx
├── pages/           ← 라우팅되는 "화면" 단위
│   ├── Home.jsx
│   ├── Whisky.jsx
│   ├── Gin.jsx
│   └── Rum.jsx
├── components/       ← 여러 화면에서 재사용되는 "부품"
│   ├── CocktailGrid.jsx
│   ├── CocktailDraw.jsx
│   ├── SpiritLoader.jsx
│   └── BottleShape.jsx
├── data/            ← JSX 없는 순수 정적 데이터
│   ├── cocktailSpirits.js
│   └── bottleShapes.js
└── utils/           ← JSX 없는 순수 로직 함수
    └── navMemory.js
```

포인트는 **"위스키/진/럼 페이지 3개를 각각 따로 만들지 않았다"**는 것이다. `Whisky.jsx`, `Gin.jsx`, `Rum.jsx`는 사실상 같은 `CocktailGrid` 컴포넌트에 다른 `props`만 넘기는 얇은 래퍼(wrapper)에 불과하다.

```jsx
// pages/Whisky.jsx
import CocktailGrid from '../components/CocktailGrid.jsx'
import { SPIRIT_COCKTAILS } from '../data/cocktailSpirits.js'

const Whisky = () => (
  <CocktailGrid
    spirit="whisky"
    title={SPIRIT_COCKTAILS.whisky.title}
    cocktailNames={SPIRIT_COCKTAILS.whisky.cocktails}
  />
)
```

`data/cocktailSpirits.js`에는 위스키·진·럼 각각 12개씩 칵테일 이름만 배열로 박아뒀고, `CocktailGrid`는 그 이름 배열만 받아서 API 호출부터 그리드 렌더링까지 전부 담당한다. 즉 **페이지가 3개로 늘어나도 실제로 작성하는 로직은 하나**다.

| 용도 | 예시 |
|---|---|
| API로 받아온 데이터 저장 | `const [cocktails, setCocktails] = useState([])` |
| 화면에 뭘 보여줄지 결정하는 상태 | `const [isLoading, setIsLoading] = useState(true)`, `const [phase, setPhase] = useState('idle')` |
| 사용자 상호작용 추적 | `const [hovered, setHovered] = useState(null)` |

한 컴포넌트 안에 `useState`가 3개 이상 얽히기 시작하면(`CocktailDraw.jsx`의 `cards`, `picked`, `phase`처럼) 각각을 따로 관리하는 대신 `useReducer`로 묶는 게 자연스러운 다음 단계다. `data/`, `utils/`처럼 JSX가 없는 순수 함수·상수를 컴포넌트 파일과 분리해둔 덕분에, 칵테일 목록을 추가하거나 병 모양 SVG 좌표를 수정할 때 `CocktailGrid.jsx`나 `BottleShape.jsx` 같은 컴포넌트 코드는 건드릴 필요가 없었다.

### TheCocktailDB 무료 API로 칵테일 레시피 가져오기

자체 DB 없이 **[TheCocktailDB](https://www.thecocktaildb.com/)라는 무료 공개 칵테일 레시피 API**를 붙여서 실제 레시피 데이터(재료, 계량, 제조법, 사진)를 전부 가져왔다. 회원가입이나 API 키 발급 없이 테스트용 키 `1`로 바로 호출이 가능하다.

```js
const fetchCocktail = async (name) => {
  const res = await fetch(
    `https://www.thecocktaildb.com/api/json/v1/1/search.php?s=${encodeURIComponent(name)}`
  )
  const data = await res.json()
  return data.drinks ? data.drinks[0] : null
}
```

까다로운 부분은 이 API가 재료를 `strIngredient1`~`strIngredient15`, 계량을 `strMeasure1`~`strMeasure15`처럼 **번호가 매겨진 15개의 개별 필드**로 흩어서 내려준다는 점이다(재료가 3개뿐이어도 필드 자체는 15개까지 존재하고 나머지는 `null`). 배열이 아니라 이런 구조이므로 반복문으로 직접 모아서 리스트로 바꿔야 한다.

```js
const getIngredients = (drink) => {
  const list = []
  for (let i = 1; i <= 15; i++) {
    const ingredient = drink[`strIngredient${i}`]
    const measure = drink[`strMeasure${i}`]
    if (!ingredient) continue
    list.push(`${measure ? measure.trim() : ''} ${ingredient}`.trim())
  }
  return list
}
```

한 페이지(예: 위스키 12종)를 렌더링할 때는 이름 12개를 순서대로 하나씩 기다리는 대신, `Promise.all`로 12개의 `fetch`를 동시에 날려 병렬로 가져온다.

```js
Promise.all(cocktailNames.map(fetchCocktail)).then((results) =>
  setCocktails(results.filter(Boolean))
)
```

실제로 연동해보면서 발견한 제약도 하나 있었다. 이름으로 검색하는 `search.php`는 결과를 전부 돌려주지만, 재료로 필터링하는 `filter.php`는 **무료 테스트 키에서는 미리보기용으로 딱 1개만** 돌려준다 — 같은 API라도 엔드포인트별로 무료 티어 제한이 다를 수 있다는 걸 직접 확인한 셈이다. 결국 이 앱에서는 재료 필터링(`filter.php`) 대신 이름 검색(`search.php`)만 써서 `SPIRIT_COCKTAILS`에 미리 정해둔 12개 이름으로 데이터를 채웠다.

### 페이지 이동 간 상태 공유: 라우터 state vs 순수 JS 변수

`<Link to="/" state={{ from: 'whisky' }}>`처럼 넘긴 `state`는 브라우저의 히스토리 엔트리에 저장된다. 문제는 이 히스토리 state가 **새로고침해도 사라지지 않는다**는 점이다. "새로고침하면 무조건 초기 상태로 리셋되길" 원하는 로직에는 맞지 않는다.

해결책은 라우터 state 대신 **순수 JS 모듈 변수**를 쓰는 것이다.

```js
// navMemory.js
let lastSpirit = null
export const setLastSpirit = (spirit) => { lastSpirit = spirit }
export const getLastSpirit = () => lastSpirit
```

이 변수는 SPA 안에서 페이지를 이동하는 동안에는(JS 런타임이 계속 살아있으므로) 값이 유지되지만, 새로고침하면 스크립트가 처음부터 다시 실행되면서 자동으로 `null`로 초기화된다.

| | 지속 범위 | 새로고침 시 |
|---|---|---|
| `<Link state={...}>` | 브라우저 히스토리 엔트리 | **유지됨** |
| 모듈 스코프 `let` 변수 | 현재 JS 런타임(탭이 살아있는 동안) | **초기화됨** |

실제 앱에서는 이 값을 이용해 홈 화면 문구를 바꾼다. 레시피 페이지에서 "처음으로"를 눌러 돌아오면 방금 보고 있던 술 종류(`spirit`)를 뺀 나머지 둘 중 하나를 랜덤으로 추천하고, 주소창으로 바로 들어온 첫 방문이면 고정 문구를 보여준다.

```js
const pickPrompt = (cameFrom) => {
  if (!cameFrom) return DEFAULT_PROMPT
  const others = SPIRIT_ORDER.filter((key) => key !== cameFrom)
  const picked = others[Math.floor(Math.random() * others.length)]
  return SPIRIT_PROMPTS[picked]
}
```

### FLIP 애니메이션과 CSS 3D 카드 뒤집기

카드가 화면의 원래 자리에서 중앙으로 자연스럽게 확대 이동하는 애니메이션은 **FLIP 기법**(First-Last-Invert-Play)의 축약 버전으로 만들 수 있다. 핵심은 애니메이션을 시작하기 직전의 실제 화면 좌표를 `getBoundingClientRect()`로 먼저 재두는 것이다.

```jsx
const handlePick = (card, event) => {
  const rect = event.currentTarget.getBoundingClientRect()
  setPicked({ ...card, rect }) // 클릭 시점의 실제 위치를 기억
}

// 렌더링
<div
  className={isRevealed ? 'draw-card--floating is-revealed' : 'draw-card--floating'}
  style={isRevealed ? undefined : { top: picked.rect.top, left: picked.rect.left, width: picked.rect.width, height: picked.rect.height }}
/>
```

처음엔 인라인 스타일로 "원래 있던 자리 그대로"에 카드를 띄우고, 그다음 프레임에 `isRevealed` 클래스를 붙여 CSS(`top: 50%; left: 50%; transform: translate(-50%, -50%)`)가 적용되게 하면, `transition`이 알아서 "원래 위치 → 중앙"으로 부드럽게 보간해준다. 카드가 몇 번째 줄·칸에 있었는지 미리 계산할 필요 없이 어디서 클릭하든 항상 그 자리에서 자연스럽게 움직인다.

```
1. First : 애니메이션 시작 전 요소의 실제 위치를 getBoundingClientRect()로 측정
2. Last  : 목표 상태(클래스 등)를 적용해 최종 위치/크기를 결정
3. Invert: (여기서는 "시작 위치를 인라인 스타일로 고정 → 다음 틱에 목표 클래스 적용"으로 단순화)
4. Play  : CSS transition이 나머지를 부드럽게 처리
```

카드 3D 뒤집기는 순수 CSS 세 속성 조합으로 구현된다.

```css
.card {
  perspective: 700px; /* 부모: 원근감(입체감) 정도 */
}
.card__inner {
  transform-style: preserve-3d; /* 자식들의 3D 배치를 유지 */
  transition: transform 0.6s ease;
}
.card__inner.is-flipped {
  transform: rotateY(180deg);
}
.card__face {
  position: absolute;
  inset: 0;
  backface-visibility: hidden; /* 뒷면을 봤을 때는 숨김 */
}
.card__back {
  transform: rotateY(180deg); /* 뒷면은 미리 180도 돌려둠 */
}
```

앞면·뒷면을 같은 자리에 겹쳐두고 뒷면만 미리 180도 돌려두면, 부모가 `rotateY(180deg)`로 회전할 때 앞면은 사라지고(`backface-visibility: hidden`) 뒷면이 정방향으로 나타나는 것처럼 보인다.

| 속성 | 역할 |
|---|---|
| `perspective` | 부모에 원근감 부여 (숫자가 작을수록 입체감 과장) |
| `transform-style: preserve-3d` | 자식 요소들의 3D 배치를 평면화하지 않고 유지 |
| `backface-visibility: hidden` | 회전해서 뒤로 넘어간 면은 안 보이게 |
| `rotateY(180deg)` | 실제 회전 애니메이션 |

`CocktailDraw.jsx`에서는 카드 앞면(`BottleFace`, 물음표가 그려진 병 실루엣)과 뒷면(칵테일 사진+이름)을 이 방식으로 겹쳐뒀다가, API 응답이 도착하면(`picked.drink`) `is-flipped` 클래스를 붙여 뒤집는다.

### 최소 로딩 시간 보장

API 응답이 오는 즉시 `isLoading`을 꺼버리면 로딩 애니메이션이 너무 빨리 사라진다. API 요청과 "최소 대기 시간"을 `Promise.all`로 묶으면 최소 표시 시간을 보장할 수 있다.

```js
const wait = (ms) => new Promise((resolve) => setTimeout(resolve, ms))

Promise.all([
  fetchCocktail(name),
  wait(1500), // 최소 1.5초
]).then(([drink]) => setIsLoading(false))
```

`Promise.all`은 **가장 오래 걸리는 것**이 끝나야 완료되므로, API가 200ms만에 응답해도 최소 1.5초는 `SpiritLoader`(병 아이콘 + "OO 따르는 중..." 문구) 화면이 유지되고, API가 3초 걸리면 3초를 그대로 기다린다.

## 도구 & 환경

### 빌드 결과물 배포: npm run build → Netlify

my-app2를 실제로 배포까지 해봤다. `npm run build`를 실행하면 JSX가 전부 순수 HTML/CSS/JS로 변환된 정적 파일이 `dist/` 폴더에 생성되고, 이 폴더를 그대로 Netlify에 드래그 앤 드롭하거나 `netlify deploy --prod --dir=dist` 명령으로 배포했다.

## 핵심 정리
- `react-router-dom`은 URL을 브라우저 안에서만 관리되는 가짜 경로로 다루고, 주소가 바뀌면 화면 일부만 바꿔치기하는 SPA 라우팅을 구현한다
- `pages/`(라우팅 화면), `components/`(재사용 UI), `data/`(순수 정적 데이터), `utils/`(순수 로직)로 역할을 나누면, 위스키/진/럼 페이지 3개를 각각 만들지 않고 `CocktailGrid` 하나에 props만 다르게 넘겨서 재사용할 수 있다
- 무료 API(TheCocktailDB)는 회원가입 없이 테스트 키로 바로 붙일 수 있지만, `strIngredient1~15`처럼 번호 매김 필드 구조나 엔드포인트별 응답 제한(`search.php` vs `filter.php`)처럼 실제로 붙여봐야 알 수 있는 제약이 있다
- 페이지 간 상태 공유는 "새로고침해도 유지"가 필요하면 라우터 `state`, "새로고침 시 리셋"이 필요하면 모듈 스코프 `let` 변수를 쓴다
- FLIP 애니메이션은 `getBoundingClientRect()`로 시작 위치를 재고 CSS transition으로 자연스럽게 이동시키는 기법이며, 3D 카드 뒤집기는 `perspective` + `preserve-3d` + `backface-visibility` 조합으로 만든다
- `Promise.all([요청, wait(ms)])`로 API가 아무리 빨라도 최소 로딩 시간을 보장할 수 있고, `npm run build`로 뽑은 `dist/`는 Node.js 서버 없이 정적 호스팅에 그대로 배포된다
