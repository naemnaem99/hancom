## 도구 & 환경

### groq-chat을 Vercel에 배포하기 — 겪은 시행착오

어제(12일차) 만든 Groq 챗봇을 `hancom` 모노레포 안(`07_groq`)에서 바탕화면의 독립 폴더(`groq-chat`)로 분리했다. 이유는 간단한데, Vercel에 배포하려면 **이 프로젝트 하나만 담은 별도 GitHub 저장소**가 있어야 하기 때문이다. 모노레포 전체를 배포 대상으로 넘길 수는 없다.

분리 직후 `git push`를 했는데 뭔가 이상했다. 확인해보니 **연결된 원격 저장소(`origin`)가 여전히 기존 `hancom` 레포를 가리키고 있었다.** 폴더는 옮겼지만 git 설정까지 완전히 새로 정리하지 않았던 게 원인이다. 그래서 새 GitHub 저장소(`naemnaem99/groq-chat`)를 만들고 origin을 그쪽으로 다시 연결한 뒤에야 제대로 push가 됐다.

> 정확히는 "깃 연결이 하나의 레포에만 되는" 게 아니라, `origin`이라는 이름표가 **한 번에 하나의 주소만** 가리킬 수 있는 것이다. 여러 원격 저장소를 동시에 등록하는 것도 가능하지만(`git remote add 다른이름 주소`), 보통은 `origin` 하나만 쓰기 때문에 폴더를 복사해왔을 때 "예전 주소가 그대로 남아있는" 문제가 생기기 쉽다.

```bash
git remote -v                        # 지금 origin이 어디를 가리키는지 먼저 확인
git remote set-url origin 새주소      # 다른 곳을 가리키고 있었다면 교체
```

새 프로젝트를 시작할 때 기존 폴더를 복사해왔다면, `git remote -v`로 연결부터 확인하는 습관이 필요하다는 걸 배웠다.

### 로컬 서버에서 Vercel 서버리스 함수로

로컬에서는 Express 서버(`server/index.js`)가 상시 켜져 있는 방식으로 잘 작동했다. 그런데 Vercel은 그런 "항상 켜져 있는 서버"를 그대로 못 돌린다. 대신 **`api/` 폴더 안의 파일 하나하나를 "요청이 올 때만 잠깐 실행되는 서버리스 함수"로 인식**하는 구조다. 그래서 Express가 하던 일(Groq API에 대신 요청 보내기)을 `api/chat.js`라는 서버리스 함수 형태로 새로 옮겨 적었다.

```js
// api/chat.js — Vercel 서버리스 함수
module.exports = async (req, res) => {
  if (req.method !== 'POST') {
    res.status(405).json({ error: 'Method not allowed' })
    return
  }

  const key = process.env.GROQ_API_KEY
  const { prompt } = req.body

  const groqRes = await fetch('https://api.groq.com/openai/v1/chat/completions', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer ' + key
    },
    body: JSON.stringify({
      model: 'llama-3.1-8b-instant',
      messages: [{ role: 'user', content: prompt }]
    })
  })

  const data = await groqRes.json()
  res.status(200).json({ reply: data.choices?.[0]?.message?.content || '(응답없음)' })
}
```

| | 로컬 Express 서버 | Vercel 서버리스 함수 |
|---|---|---|
| 실행 방식 | 상시 켜져 있음 | 요청 올 때만 잠깐 실행 |
| 파일 위치 | 자유(`server/index.js` 등) | `api/` 폴더 규칙을 따름 |
| API 키 위치 | `.env` 파일 | Vercel 환경변수 설정 |

프로토타입은 로컬 Express로 빠르게 테스트하고, 배포 단계에서 서버리스 함수 형태로 옮기는 흐름은 개인 프로젝트에서 흔한 패턴이다.

### API 키, 어디에 두고 어떻게 지켰나

- 로컬 테스트 때는 `server/.env`에 `GROQ_API_KEY=실제키값`을 적어두고, Express가 `process.env.GROQ_API_KEY`로 읽었다.
- `.env`는 `.gitignore`에 등록돼 있어서 **GitHub에는 절대 올라가지 않는다** (`git ls-files`로 직접 확인해봐도 추적되지 않았다). 즉 Vercel이 GitHub 저장소만 보고 배포해도, 그 안에는 키 값이 아예 없다.
- 그래서 **Vercel 대시보드(프로젝트 설정 → Environment Variables)에 `GROQ_API_KEY`를 직접 등록**했다. 이렇게 해두면 Vercel이 `api/chat.js`를 실행할 때 그 값을 `process.env.GROQ_API_KEY`로 자동으로 넣어준다 — 로컬의 `.env`와 같은 역할을 Vercel에서는 "환경변수 설정"이 대신하는 것이다.

결과적으로 키는 **① 로컬 `.env`(로컬 테스트용) ② Vercel 환경변수(배포용)** 두 군데에만 존재하고, 깃 저장소 안에도 브라우저로 전달되는 코드 안에도 절대 들어가지 않는다.

### GitHub 연동 — 그 다음부턴 push=배포

GitHub 저장소와 Vercel 프로젝트를 한 번 연결해두면, 그 뒤로는 별도의 "배포" 행위가 필요 없다.

```
코드 수정 → git commit → git push origin main
   ↓ (GitHub에 push된 걸 Vercel이 감지)
Vercel이 자동으로 새 버전 빌드 + 배포
   ↓
라이브 URL이 몇 초 안에 새 코드로 교체됨
```

실제로 "챗봇 디자인 수정하고 푸시해줘"라고 지시하고 push가 성공하니, 곧바로 Vercel 프로젝트도 새 디자인으로 배포됐다. Vercel에 직접 API 요청을 보내는 게 아니라, **평소처럼 git push만 하면** Vercel이 알아서 감지해서 재배포하는 구조다.

### Claude로 웹페이지 만들기 — 단계별로 비교해보며 느낀 것

같은 "계산기 웹페이지 만들기"를 조건만 바꿔가며 단계별로 시도해봤다.

| 시도 | 체감 결과 |
|---|---|
| Plan 없이 그냥 요청 | 나쁘지 않음, 그러나 남들과 비슷한 결과물 |
| Plan 모드 + 방향성 질문 | 질문이 기본적이라 크게 안 벗어남 |
| `frontend-design` 스킬 | 조금 더 나음, "한 방에 완성"은 아님 |
| 스킬 + 타 사이트 design.md | 새로 만드는 게 아니라 "가져다 쓰는" 것 |
| 하위 모델(Haiku)로 시도 | 모델 한계는 그대로 드러남 |

이 순서대로 겪어보니, "도구(스킬/플랜/참고자료)가 좋아질수록 결과가 좋아진다"기보다는 **"내가 원하는 바를 얼마나 구체적으로 알고 지시하느냐"가 결과 품질의 진짜 상한선**이라는 걸 체감했다. 도구는 그 상한선 안에서 도달 속도를 높여줄 뿐이다.

핵심 결론은 두 가지다.

- 경험(다양한 디자인을 직접 본 것)이 쌓이면 → 클로드가 묻기 전에 내가 먼저 구체적으로 지시할 수 있다.
- 경험이 부족하면 → 클로드에게 "세부적으로 캐물어 달라"고 먼저 요청해서, 나조차 몰랐던 니즈를 대화로 끌어내야 한다.

레퍼런스를 많이 본 사람일수록 "이런 느낌으로"가 아니라 "그리드는 12컬럼, 포인트 컬러는 채도를 낮춘 코랄"처럼 구체적으로 요청할 수 있다. 경험이 부족한 단계에서는 오히려 "질문을 많이 받는 것" 자체가 실력을 대신할 수 있는 전략이 된다. (다만 "클로드가 내 니즈를 잘 캐묻도록 유도하는 방법"은 아직 답을 못 찾았고, 계속 고민해볼 주제로 남겨둔다.)

모델 선택에 대해서도 나름의 결론을 냈다.

| 예산 상황 | 전략 |
|---|---|
| 넉넉함 | 고민 없이 최상위 모델(Opus 등)로 전 과정 진행 |
| 빠듯함 | 모델별 장단점을 파악해서, 프로젝트 단계별(기획/구현/디테일)로 적절한 등급의 모델을 배분 |

예산이 제한된 팀에서는 초안/반복 실험 단계는 저렴한 모델로 빠르게 돌리고, 최종 완성도를 다듬는 단계에서만 상위 모델로 전환하는 "모델 티어링" 전략을 쓰기도 한다.

## 핵심 정리
- 프로젝트를 독립 저장소로 분리할 때는 `git remote -v`로 `origin` 연결부터 확인하자 — 예전 주소가 그대로 남아있기 쉽다
- Vercel은 상시 서버가 아니라 `api/` 폴더의 서버리스 함수로 요청을 처리하므로, 로컬 Express 코드를 그 구조에 맞게 옮겨야 한다
- 비밀 API 키는 로컬 `.env` + Vercel 환경변수 두 곳에만 두고, 깃 저장소와 클라이언트 코드에는 절대 노출하지 않는다
- GitHub 저장소와 Vercel을 한 번 연동해두면 이후로는 커밋+푸시만으로 자동 재배포된다
- Claude로 웹페이지를 만들 때 결과 품질을 가르는 진짜 변수는 도구(Plan/스킬/design.md/모델)가 아니라 **내가 원하는 바를 얼마나 구체적으로 아는가**이며, 경험이 부족할 땐 클로드가 세부적으로 되묻도록 유도하는 게 대안이다
