## 도구 & 환경

### feel 프로젝트로 본 비효율의 정체

"feel"이라는 감정 배출 웹앱을 Claude와 함께 만들면서, 요청이 잘게 쪼개져서 왕복이 많이 늘어난 경험을 되짚어봤다. 문제는 모든 반복이 다 나쁜 게 아니라는 점이다. 어떤 반복은 원래 그렇게 갈 수밖에 없는 정상적인 과정이었고, 어떤 반복은 애초에 한 번에 정리했으면 안 겪어도 됐을 비효율이었다.

실제로 있었던 반복 세 가지:

1. **같은 CSS 버그를 세 번 만남** — `position: fixed`로 고정한 드래그 요소는, 조상 요소 중 하나라도 `transform`이나 `perspective`가 걸려 있으면 좌표 기준(containing block)이 그 조상으로 바뀌어버린다. body의 `perspective` → `.cards`의 `transform` 순으로, 매번 새로운 조상에 `transform`을 걸었다가 똑같은 이유로 요소가 사라지는 버그를 세 번이나 반복했다.
2. **문구/버튼 텍스트를 한 줄씩 여러 메시지로 나눠서 요청** — 버튼 라벨 3단계, placeholder 3종, 초기 프롬프트 문구가 전부 따로따로 왔다.
3. **핵심 컨셉이 뒤늦게 확정** — 처음엔 기분 카드 4개(감정 도트 패턴)로 만들었다가, 최종적으로는 카드 1개짜리 "바구니" 컨셉으로 완전히 재해석되면서 앞의 작업 대부분이 버려졌다.

### 정상적인 시행착오 vs 피할 수 있는 비효율

기준은 간단하다. **직접 보기 전엔 판단할 수 없는 것**은 미리 다 정해놓고 시작해도 결국 보고 나서 뒤집었을 것들이라 비효율이 아니다. 반면 **구조적 규칙**이나 **핵심 컨셉**을 뒤늦게 알게 되어 같은 실수를 반복한 경우는 애초에 피할 수 있었던 비효율이다.

| 구분 | 예시 | 처리 방식 |
|---|---|---|
| 시각적 판단이 필요한 것 | 이모지/픽셀 디자인, 애니메이션 느낌 | 만들어보고 반복 수정 — 정상적인 과정 |
| 구조적 규칙 | 드래그 요소는 항상 `fixed` 유지, 조상에 `transform` 금지 | 처음에 요구사항으로 명시하면 같은 버그 반복 예방 |
| 반복되는 텍스트/문구 | 버튼 라벨 3단계, placeholder 3종 | 표로 한 번에 정리해서 요청 |
| 핵심 컨셉 | "바구니로 텍스트를 담는다" | 먼저 확정하면 그 이전 단계 작업(카드 4개 등)을 생략 가능 |

정리하면, **큰 구조와 상호작용 규칙은 한 번에 몰아서 요청**하고, **색·크기·문구 같은 디테일은 결과물을 보고 나서 하나씩 다듬는 식**으로 나눠서 요청하는 게 가장 효율적이다.

### 한 번에 정리해서 던지는 프롬프트 예시

같은 프로젝트를 다시 시작한다면 아래 정도의 프롬프트 하나로 시작했을 것이다.

```
"feel"이라는 감정 배출 웹앱을 만들자. 컨셉: 배출 → 선택 → 소각.

1. input+버튼으로 감정 입력 → 클릭 3번에 걸쳐 문구/버튼 텍스트가 단계적으로 바뀌고
   (초기/좋은감정/나쁜감정), 3번째 클릭에 리셋되면서 화면 중앙에 바구니 등장.
   입력한 텍스트는 눈처럼 화면에 흩날림.
2. 바구니: 위쪽 뚫린 픽셀아트 상자, 자유롭게 드래그 가능(항상 fixed 포지션 유지,
   조상에 transform 금지). 입구(상단중앙 1/3)에 닿는 낙하 텍스트만 선택적으로 캐치.
3. 바구니를 처음 만지면 하단에 불꽃 등장, 바구니를 불로 끌고 가면 소각 후 새 바구니 재생성.
4. 입력 기록은 localStorage에 3개 사이클 단위로 저장, 새로고침 시 마지막 사이클만 재생.
스타일: 미니멀 흑백 픽셀아트.
```

이 정도만 있었어도 최종 결과물의 80%는 훨씬 적은 왕복으로 만들 수 있었을 것이다. 핵심은 "모든 디테일을 미리 정하기"가 아니라, **나중에 알아도 되는 것과 미리 알아야 하는 것을 구분하는 것**이다.

## 코딩 학습

### feel 프로젝트에 쓴 HTML/CSS/JS 기술

"feel"(감정 배출 웹앱)을 만들면서 실제로 사용한 HTML/CSS/JS 기법을 정리했다.

**HTML**은 시맨틱 태그 없이 순수 `div` 여러 개(`#prompt`, `#controls`, `#cards`, `#basketHint`, `#fire`, `#fireHint`)만 뼈대로 두고, 내용은 전부 JS가 채워 넣는 구조다. `<script src="scripts/main.js">`를 `</body>` 바로 앞에 둬서 DOM이 다 그려진 뒤 스크립트가 실행되게 했다.

**CSS**에서는 아래 기법들을 썼다.

| 기법 | 어디에 썼나 |
|---|---|
| `position: fixed` + `inset: 0` | 카드/눈송이 텍스트를 화면(뷰포트) 전체 기준으로 고정 배치 |
| `pointer-events: none` / `auto` | `.cards` 컨테이너는 클릭이 통과되게, 그 안의 실제 카드만 클릭 가능하게 분리 |
| `@keyframes` (`cardIn`, `fall`) | 카드 등장 애니메이션, 텍스트가 눈처럼 떨어지는 애니메이션 |
| `box-shadow`를 여러 개 겹쳐서 픽셀 그림 그리기 | 화살표 아이콘을 이미지 파일 없이 CSS만으로 표현 |
| `touch-action: none` | 드래그 중 모바일에서 스크롤/줌이 같이 발생하는 걸 방지 |
| `@media (max-width: 600px)` | 좁은 화면에서 카드 크기 축소 |

**JS**에서는 아래 기법들을 썼다.

| 기법 | 어디에 썼나 |
|---|---|
| `localStorage` + `JSON.stringify`/`parse` | 입력 기록을 3개 단위로 묶어 저장, 새로고침 시 마지막 묶음만 재생 |
| `document.createElement` + `appendChild` | 카드, 화살표 아이콘, 눈송이 텍스트를 전부 동적으로 생성 |
| Pointer Events (`pointerdown`/`pointermove`/`pointerup`) | 마우스와 터치를 하나의 이벤트 체계로 통합해서 드래그 구현 |
| 사각형 충돌 판정 함수 `overlaps(a, b)` | "카드가 불에 닿았는지"와 "낙하 텍스트가 바구니 입구에 닿았는지" 두 곳에서 재사용 |
| `requestAnimationFrame` 반복 루프 | 매 프레임마다 낙하 텍스트와 바구니 입구의 충돌을 계속 검사 |
| 클릭 횟수 상태(`clickCount`)로 단계 전환 | 버튼 하나로 안내 문구·placeholder·버튼 텍스트를 세트로 3단계 전환 |
| `animationend` 이벤트로 요소 재활용 | 떨어지는 텍스트가 바닥에 닿으면 다시 랜덤 위치에서 재시작 |

### 복습: 왜 이렇게 짰을까

`.cards`에는 `pointer-events: none`을, 그 안의 카드에는 다시 `auto`를 걸었다. `.cards`는 화면 전체를 덮는 `fixed` 레이어라서 그대로 두면 아래 있는 입력창·버튼 클릭을 다 막아버리기 때문이다. 컨테이너는 클릭을 "투과"시키고, 실제 카드 요소만 다시 클릭 가능하게 살려서 카드 바깥 영역은 다른 UI를 그대로 조작할 수 있게 했다.

충돌 검사(`checkCatches`)는 `setInterval`이 아니라 `requestAnimationFrame`으로 반복 호출했다. `requestAnimationFrame`은 브라우저가 화면을 실제로 다시 그리는 타이밍(보통 초당 60회)에 맞춰 실행되고, 탭이 비활성화되면 자동으로 쉬어서 배터리/성능에 유리하다. 애니메이션 중인 요소의 위치를 매 프레임 검사해야 하는 충돌 판정에는 고정된 시간 간격인 `setInterval`보다 자연스럽게 맞는다.

`overlaps()` 하나로 두 가지 다른 충돌(카드-불, 텍스트-바구니)을 처리할 수 있었던 이유는, 두 상황 모두 결국 "사각형 두 개가 겹치는가"라는 같은 문제였기 때문이다. `getBoundingClientRect()`로 구한 좌표(`left/right/top/bottom`)만 넘기면 어떤 요소끼리든 재사용 가능했다. DOM 요소가 아니라 좌표값을 인자로 받게 만든 게 재사용의 핵심이다.

```js
const overlaps = (a, b) => (
    a.left < b.right &&
    a.right > b.left &&
    a.top < b.bottom &&
    a.bottom > b.top
);
```

`localStorage`에는 배열 안에 배열(`groups` 안의 `lastGroup`) 구조로 저장했다. 한 사이클(3번 입력)을 하나의 묶음으로 관리하기 위해서다. `lastGroup`이 꽉 차면 새 배열을 만들어 `groups`에 추가하는 식으로, 새로고침 후에는 가장 최근 묶음(`groups[groups.length - 1]`)만 다시 재생할 수 있다.

마우스 이벤트 대신 Pointer 이벤트를 쓴 이유는, `mousedown`/`touchstart`처럼 입력 방식별로 이벤트를 따로 처리할 필요 없이 마우스·터치·펜 입력을 하나의 API로 통합해서 다룰 수 있기 때문이다. 드래그 로직을 한 번만 작성해도 데스크톱/모바일 모두 동작한다.

## 핵심 정리
- 시각적 디테일(이모지, 애니메이션 느낌)은 반복 수정이 당연한 정상 과정
- 구조적 규칙(예: fixed 요소 조상에 transform 금지)은 처음에 명시해야 같은 버그를 반복하지 않음
- 핵심 컨셉이 늦게 확정되면 그 전 작업이 통째로 버려질 수 있으니 최대한 먼저 확정
- 반복되는 문구/라벨류는 한 메시지에 표로 정리해서 한 번에 요청
- "큰 구조는 몰아서, 디테일은 보고 나서" 원칙으로 왕복 횟수를 크게 줄일 수 있음
- 화면 전체를 덮는 레이어는 `pointer-events: none` + 자식만 `auto`로 클릭 영역을 세밀하게 제어
- 반복되는 애니메이션 프레임 검사는 `setInterval`보다 `requestAnimationFrame`이 적합
- 충돌 판정 함수는 DOM 요소가 아니라 좌표(rect)를 받게 만들면 여러 상황에 재사용 가능
- 드래그 기능은 마우스/터치를 따로 안 다루고 Pointer Events로 통일하는 게 유리
