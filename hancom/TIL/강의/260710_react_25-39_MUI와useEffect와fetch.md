# [React 25-39번] 조건부 렌더링, MUI, useEffect, fetch 정리

> 작성: 2026-07-10 | 수정: 2026-07-14

## Q&A

**Q. `<Tag tags={"list"}/>`처럼 문자열로 감싸면 왜 실제 배열이 전달 안 되나?**

JSX에서 `{}`는 "안에 있는 걸 JS 표현식으로 평가해라"는 뜻이다. `{"list"}`는 `list`라는 **문자열 리터럴**이고, `{list}`는 `list`라는 **변수 참조**다. 실제 배열 변수를 넘기려면 따옴표 없이 `{list}`로 써야 한다.

**Q. `.map()`으로 배열을 렌더링할 때 `key`는 왜 꼭 필요한가?**

React가 리스트의 각 항목을 구분해서 추적하기 위한 고유 식별자다. `key`가 없으면 리스트가 바뀔 때(추가/삭제/순서변경) React가 어떤 항목이 바뀌었는지 정확히 몰라서 전체를 다시 그리거나 엉뚱한 항목이 갱신될 수 있다. 배열 인덱스보다는 데이터 자체의 고유값(`id`)을 쓰는 게 안전하다.

```jsx
{users.map((u) => (
    <li key={u.id}>{u.name}</li>
))}
```

**Q. controlled input(`value` + `onChange`)이 뭔가, 왜 필요한가?**

input의 현재 값을 브라우저(DOM)가 아니라 **React state가 관리**하는 방식이다. `value={state}`로 화면에 보여줄 값을 고정하고, `onChange`에서 `setState(e.target.value)`로 타이핑할 때마다 state를 갱신한다. 이렇게 해야 입력값을 JS 코드에서 실시간으로 읽고 검증/가공할 수 있다. `value`만 주고 `onChange`가 없으면 입력이 아예 안 먹는 읽기 전용 인풋이 된다.

**Q. MUI `Button`/`TextField`를 쓸 때 `sx` prop이 뭔가?**

MUI 컴포넌트 전용 스타일 지정 방식. 일반 CSS 대신 객체로 스타일을 넘기는데, 숫자만 쓰면 MUI의 spacing 단위(기본 8px 배수)로 자동 변환되고, 테마 색상 토큰(`primary.main` 등)이나 `&:hover` 같은 가상 선택자도 그대로 객체 키로 쓸 수 있다.

```jsx
<Button sx={{ mt: 2, '&:hover': { backgroundColor: 'primary.dark' } }}>제출</Button>
```

**Q. `AppBar`와 `Toolbar`를 왜 같이 쓰나?**

`AppBar`는 화면 상단에 고정되는 바 자체(배경색, 그림자, position 등)를 담당하고, `Toolbar`는 그 안의 내용물을 가로로 배치하고 좌우 여백/높이를 표준 규격으로 맞춰주는 레이아웃 컨테이너다. `AppBar` 혼자 쓰면 안의 요소들이 여백 없이 다닥다닥 붙는다.

**Q. `Tabs`/`Tab` 클릭 시 파란 밑줄이 이동하고 배경색이 바뀌는 건 MUI 기본값인가?**

맞다, `Tabs`의 기본 테마 동작이다. 밑줄 색은 `TabIndicator`의 색상(`sx={{ '& .MuiTabs-indicator': { backgroundColor: '...' } }}`), 선택된 탭 배경/글자색은 `Tab`의 `sx`나 `selected` 상태 스타일로 커스텀할 수 있다.

**Q. 메뉴(Tab)를 클릭했을 때 탭마다 다른 콘텐츠를 보여주려면?**

탭이 선택된 상태(`value`)를 `Menu` 내부가 아니라 **부모(App)로 끌어올려야(state lifting)** 한다. `Menu`는 자체 `useState` 대신 `value`/`onChange`를 props로 받는 controlled 컴포넌트로 바꾸고, `Content`는 `tab` 값을 props로 받아서 `PAGES[tab]`처럼 매핑된 내용을 렌더링한다.

```jsx
// App.jsx
const [tab, setTab] = useState(0)
<Menu value={tab} onChange={(_, v) => setTab(v)} />
<Content tab={tab} />
```

**Q. `flexGrow`를 셋에 다 줬는데 왜 정확히 3등분이 안 되나?**

`flexGrow`만 주면 `flex-basis`는 기본값 `auto`(콘텐츠 크기)로 남는다. 그래서 빈 `Box`(내용 없음, 크기 0에 가까움)와 버튼이 든 `Box`(실제 크기 있음)가 같은 비율로 늘어나긴 해도 시작 크기가 다르니 결과가 정확히 1:1:1이 되지 않는다. `flex: 1` 축약형을 쓰면 `flex-basis: 0%`로 시작점을 통일시켜서 진짜 균등 분배가 된다.

| 속성 | flex-basis | 결과 |
|---|---|---|
| `flexGrow: 1` | `auto` (콘텐츠 크기 유지) | 시작 크기가 다르면 비율이 어긋남 |
| `flex: 1` | `0%` | 시작 크기 무시하고 완전 균등 분배 |

단, `flex: 1`은 내용이 길어지면 강제로 눌러서 줄바꿈/찌그러짐이 생길 수 있어 최소 너비가 필요한 요소에는 주의해야 한다.

**Q. `useEffect(() => {...}, [])`가 "화면 뜰 때 1번만 실행된다"는 게 무슨 뜻인가?**

컴포넌트가 처음 화면에 나타날 때(mount) 딱 한 번만 안의 코드가 실행되고, 이후 리렌더링이 일어나도 다시 실행되지 않는다는 뜻이다. `[]`(빈 배열)는 "이 값이 바뀌면 다시 실행해라"는 의존성 목록인데, 감시할 값이 없으니 처음 한 번 이후로는 재실행 조건이 없는 것. 초기 데이터 fetch, 타이머 설치처럼 "한 번만 해도 되는" 부수효과에 쓴다.

**Q. `setInterval`을 `useEffect` 안에서 쓸 때 `clearInterval`(cleanup)이 왜 필요한가?**

`useEffect`가 재실행되거나 컴포넌트가 사라질 때(unmount) 이전에 설치한 타이머를 정리하지 않으면 타이머가 계속 쌓여서 중복 실행되거나 존재하지 않는 컴포넌트를 계속 갱신하려는 메모리 누수가 생긴다. `useEffect` 안에서 함수를 `return`하면 그게 cleanup 함수로 등록되어 다음 실행 직전/unmount 시 자동 호출된다.

```jsx
useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000)
    return () => clearInterval(id)
}, [])
```

**Q. `useEffect(() => {...}, [count])`처럼 의존성 배열에 값을 넣으면?**

`count`가 바뀔 때마다(첫 렌더 포함) 이펙트가 다시 실행된다. `[]`(마운트 1회)와 아무것도 안 넣은 경우(매 렌더마다) 사이의 중간 지점으로, "이 값이 바뀔 때만 반응하고 싶다"는 경우에 쓴다.

**Q. deps 배열이 아예 없는 `useEffect` 안에서 `setState`를 호출하면 어떻게 되나?**

렌더링될 때마다 이펙트가 실행되고, 그 안에서 state를 바꾸면 다시 렌더링이 일어나 이펙트가 또 실행되는 **무한 루프**가 발생한다. React의 렌더링 중 무한루프 방지 장치는 렌더 함수 자체에서 동기적으로 setState할 때 걸리는 것이라, `useEffect`(커밋 이후 비동기 실행) 안에서는 그 가드가 작동하지 않는다. 다만 콘솔 로그가 무한히 쌓이고 CPU를 계속 점유하므로 오래 켜두면 실제로 브라우저가 버벅일 수 있다. VSCode의 ESLint(`react-hooks` 플러그인)도 이 패턴을 "무한 업데이트 체인" 경고로 잡아준다.

> **[수정: 2026-07-14]** 위 설명은 "deps 배열 자체가 없는" 경우 얘기고, `Every.jsx`는 실제로는 `useEffect(fn, [])`처럼 **빈 배열**을 쓴다. 빈 배열은 "마운트 시 딱 1번"만 실행 조건이라, 안에서 `setCount`를 불러도 이펙트 자체가 다시 실행되지 않아 **무한 루프가 아니라 딱 1번 추가 렌더링**으로 끝난다. 헷갈리기 쉬운 포인트는 "이펙트 안에서 state를 바꾸는지"가 아니라 **"deps 배열이 무엇인지"가 재실행 여부를 결정한다**는 것 — `Every.jsx`의 주석("렌더링 될 때마다 실행")은 실제 동작과 다른 오해의 소지가 있는 설명이니 주의.

**Q. fetch로 받아온 데이터의 `u.id`, `u.name`에서 `u`는 무슨 뜻이고, `id`/`name`은 왜 마음대로 못 바꾸나?**

`u`는 `.map((u) => ...)`의 콜백 매개변수 이름일 뿐이라 `k`든 뭐든 원하는 대로 바꿔도 결과는 똑같다(임의로 붙인 이름). 반면 `id`/`name`은 **fetch로 받아온 실제 API 응답 데이터에 붙어 있는 고정된 속성 이름**이라, 서버가 그 이름으로 데이터를 내려주는 이상 그대로 써야 값을 읽을 수 있다. 응답 내용은 브라우저에서 fetch 링크로 직접 접속하거나, 개발자도구 Network 탭에서 확인 가능하다(단순히 F12를 눌러 콘솔만 열어서는 안 보이고, `console.log(data)`를 코드에 넣어야 콘솔에 찍힌다).

**Q. `<li>` 안에 `<br/>`를 넣었는데 왜 항목 사이 여백이 안 생기나?**

`<br/>`는 같은 요소 **안에서** 줄바꿈만 만들 뿐, 서로 다른 `<li>` 형제 요소 사이의 간격에는 영향을 주지 않는다. 리스트 항목 사이 여백은 부모 `<ul>`에 `display: flex; flexDirection: column; gap: 16px`처럼 `gap`을 주는 방식으로 해결해야 한다.

**Q. `temp ? ... : ...`처럼 쓰면 왜 기온이 0℃일 때 "불러올 수 없음"으로 나오나?**

`0`은 JS에서 falsy 값이라 삼항 연산자 조건에서 `false`와 똑같이 취급된다. "값이 아직 없다"와 "값이 0이다"를 구분하려면 falsy 여부가 아니라 `null` 여부로 명시적으로 검사해야 한다: `temp !== null ? temp + '℃' : '불러올 수 없음'`.

## 개념 요약

### 조건부 렌더링 방식 비교

| 방식 | 문법 | 쓰는 경우 |
|---|---|---|
| `&&` | `{cond && <p/>}` | 보이거나/안 보이거나 2択 |
| 삼항 연산자 | `{cond ? A : B}` | 둘 중 하나를 항상 보여줌 |
| 객체 매핑 | `{ {a:'x', b:'y'}[type] }` | 경우의 수가 3개 이상일 때 |

**실무 예시**: 로그인 여부에 따라 "로그인" 버튼과 "로그아웃" 버튼 중 하나만 보여줄 땐 `&&`나 삼항, 주문 상태("결제대기"/"배송중"/"배송완료"/"취소")처럼 상태가 여러 개고 상태별 색상까지 정해져 있으면 객체 매핑으로 관리하는 게 조건문이 계속 늘어나는 걸 막아준다.

### controlled input

```jsx
const [name, setName] = useState('')
<input value={name} onChange={(e) => setName(e.target.value)} />
```

**실무 예시**: 로그인 폼처럼 입력 도중에 "8자 이상인지", "이메일 형식이 맞는지" 실시간으로 검증하거나, 검색창에서 타이핑할 때마다 자동완성 목록을 갱신하려면 입력값을 state로 갖고 있어야 한다. 단순히 보여주기만 하는 값이 아니라 JS 로직이 개입해야 하는 입력창은 대부분 controlled로 만든다.

### 리스트 렌더링 — `.map()`, `key`, 반복 트릭

```jsx
{users.map((u) => (
    <li key={u.id}>{u.name}</li>
))}

// 데이터 없이 정해진 횟수만큼 반복할 땐 [...Array(n)]
{[...Array(5)].map((_, i) => (
    <span key={i}>{i < score ? "⭐" : "💥"}</span>
))}
```

- `key`는 React가 리스트 각 항목을 구분·추적하는 고유 식별자. 없으면 추가/삭제/순서변경 시 엉뚱한 항목이 갱신될 수 있어 배열 인덱스보다는 `id`처럼 데이터 고유값을 쓴다.
- `.map()` 콜백 매개변수명(`u`, `k` 등)은 임의로 붙인 이름이라 뭘 쓰든 상관없지만, 실제 데이터 속성명(`id`, `name`)은 API 응답에 고정된 이름이라 그대로 써야 한다.
- `Array(5)`만 쓰면 길이만 5인 빈 슬롯이라 `.map()`이 순회하지 않는다(빈 슬롯은 건너뜀). 스프레드(`[...]`)로 실제 값이 채워진 배열로 바꿔야 정상적으로 5번 돈다.
- **주의**: 콜백 매개변수 이름을 바깥 컴포넌트/변수 이름과 겹치게 짓지 말 것.

```jsx
const Tag = ({tags}) => (
    <div>
        {tags.map((Tag) => (   // ← 컴포넌트 자신과 이름이 같은 매개변수 (섀도잉)
            <span key={Tag}>{"#" + Tag}</span>
        ))}
    </div>
)
```
콜백 매개변수가 하필 바깥 컴포넌트 이름(`Tag`)과 겹치면 그 스코프 안에서 원래 이름을 가려버린다(섀도잉). 지금은 우연히 동작해도, 나중에 콜백 안에서 진짜 `Tag` 컴포넌트를 참조해야 하면 찾기 힘든 버그가 된다.

**실무 예시**: 별점처럼 서버 데이터가 아니라 정해진 횟수만큼 반복 렌더링할 때 `[...Array(n)]`을 쓰고(페이지네이션 버튼, 스켈레톤 카드 등), 콜백 매개변수 섀도잉은 리뷰에서 자주 지적되는 실수라 ESLint `no-shadow` 규칙을 켜두는 팀도 많다.

### state 갱신 — `setState` 함수형 업데이트

```jsx
// 현재 state 값을 직접 참조
<button onClick={() => setCount(count + 1)}>{count}</button>

// 함수형 업데이트
<button onClick={() => setCount(c => c + 1)}>{count}</button>
```

`setCount(count + 1)`은 그 순간 렌더링에 캡처된 값을 기준으로 계산해서, 한 핸들러 안에서 여러 번 연달아 호출하면 둘 다 같은 값을 참조해 **1만 증가**하는 함정이 있다. `setCount(c => c + 1)`처럼 함수형으로 넘기면 React가 항상 최신 값을 `c`로 넣어 호출해주므로 여러 번 호출해도 정확히 누적된다.

**실무 예시**: 더블클릭 방지 없이 빠르게 연타되는 버튼(장바구니 담기, 좋아요)이나 비동기 콜백 안에서 이전 state를 기준으로 갱신해야 하는 경우 함수형 업데이트를 기본값으로 쓰는 게 안전하다.

### MUI 핵심 컴포넌트

| 컴포넌트 | 역할 |
|---|---|
| `AppBar` + `Toolbar` | 상단 바 + 내부 레이아웃/여백 |
| `Tabs` + `Tab` | 탭 전환 UI, `value`/`onChange`로 controlled 사용 |
| `Card` + `CardContent` | 그림자 있는 콘텐츠 박스 |
| `TextField` | 입력창 (label, variant 등 내장) |
| `sx` prop | MUI 전용 인라인 스타일. 숫자는 8px 배수, 테마 토큰/가상선택자 사용 가능 |

**실무 예시**: 관리자 대시보드나 쇼핑몰 상단 네비게이션을 만들 때 `AppBar`+`Toolbar`로 뼈대를 잡고, 카테고리 전환은 `Tabs`, 상품 카드 목록은 `Card`, 검색/로그인 입력창은 `TextField`로 조립하면 디자인 시스템 없이도 빠르게 통일된 UI를 낼 수 있다. `sx`는 컴포넌트마다 CSS 파일을 따로 만들 필요 없이 그 자리에서 미세 조정할 때 자주 쓴다.

`sx`는 **MUI 컴포넌트 전용** prop이라 일반 `<button>`/`<div>`엔 안 먹힌다. 순수 HTML 요소는 표준 `style` 속성을 그대로 쓴다.

```jsx
<button style={{ border: "2px solid #5e0f0f" }}>담기</button>   {/* 일반 태그 */}
<Button sx={{ mt: 2 }}>제출</Button>                             {/* MUI 컴포넌트 */}
```

또한 컴포넌트는 실습 폴더 번호와 상관없이 어디서든 import해서 재사용할 수 있다 — `Header`(28번)가 `Counter`(29번)를 그대로 가져다 `AppBar` 안에 끼워 넣는 식이다(`import Counter from '../29/Counter'`).

### flex 정렬

```jsx
<Box sx={{ display: 'flex' }}>
    <Box sx={{ flex: 1 }}>...</Box>   {/* flex-basis: 0%, 완전 균등 분배 */}
    <Typography sx={{ flex: 1, textAlign: 'center' }}>제목</Typography>
    <Box sx={{ flex: 1 }} />          {/* 오른쪽 균형용 빈 박스 */}
</Box>
```

**실무 예시**: 헤더에 "로고 | 메뉴 | 검색창/로그인 버튼"처럼 좌우 요소 크기가 다른데도 가운데를 정확히 맞추고 싶을 때 3분할 `flex: 1` 패턴을 쓴다. 반응형에서 화면이 좁아지면 가운데 요소가 찌그러질 수 있어 보통 `minWidth`를 같이 지정한다.

### useEffect 의존성 배열 3가지 경우

| 배열 | 실행 시점 |
|---|---|
| 없음 | 렌더링될 때마다 매번 |
| `[]` | 마운트 시 딱 1번 |
| `[값]` | 마운트 시 1번 + 그 값이 바뀔 때마다 |

```jsx
useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000)
    return () => clearInterval(id)  // cleanup: 재실행/unmount 직전 호출
}, [])
```

**실무 예시**: `[]`는 페이지 진입 시 사용자 정보나 상품 목록을 한 번만 불러올 때, `[count]`처럼 특정 값 의존은 검색어(`keyword`)가 바뀔 때마다 목록을 다시 불러오는 검색 기능에 쓴다. cleanup은 실시간 채팅방처럼 소켓 연결을 열었다가 컴포넌트를 벗어나면 반드시 끊어줘야 하는 경우에 필수다.

> **[수정: 2026-07-14]** "deps 배열이 없는" 경우(위 표의 "없음")는 안에서 setState를 부르면 정말 무한루프가 되지만, **`[]`(빈 배열)는 안에서 setState를 불러도 무한루프가 아니다** — 마운트 시 딱 1번만 실행되는 조건이라 이펙트 자체가 재실행되지 않고, 딱 1번 추가 렌더링으로 끝난다. 헷갈리기 쉬운 포인트는 "이펙트 안에서 state를 바꾸는지"가 아니라 **"deps 배열이 무엇인지"가 재실행 여부를 결정한다**는 것이다.

### fetch 기본 패턴

```jsx
const [data, setData] = useState([])
const [isLoading, setIsLoading] = useState(true)

useEffect(() => {
    fetch(URL)
        .then((res) => res.json())
        .then((json) => {
            setData(json)
            setIsLoading(false)
        })
        .catch((error) => {
            console.error(error)
            setIsLoading(false)
        })
}, [])
```

**실무 예시**: 상품 상세 페이지 진입 시 서버에서 재고/가격을 불러와 `isLoading` 동안 스켈레톤 UI를 보여주다가 데이터가 오면 교체하는 패턴이 거의 표준이다. `.catch()`는 네트워크 오류나 서버 다운 시 사용자에게 "불러올 수 없음" 같은 안내를 보여주는 데 쓴다.

- 항목 간 여백은 `<br/>`이 아니라 부모의 `gap`으로
- falsy 값(`0`, `""`)이 유효한 데이터일 수 있는 조건문은 `!== null` 등으로 명시적 검사
