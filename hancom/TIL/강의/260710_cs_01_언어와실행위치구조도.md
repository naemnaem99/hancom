# [cs 01] 언어/프레임워크가 어디서 실행되는지 구조도로 정리

> 작성: 2026-07-10

## Q&A

**Q. HTML/CSS/JS까지는 이해했는데, 그 다음 React·Node.js·Java·C는 왜 이해가 안 될까?**

HTML/CSS/JS는 전부 **브라우저**라는 같은 장소에서 실행되는 삼총사라 자연스럽게 하나로 묶여서 이해된다. 반면 React·Node.js·Java·C는 "다음 단계"가 아니라 **서로 다른 위치에서 실행되는 완전히 다른 갈래**라서, HTML/CSS/JS와 같은 선상에 놓고 이해하려 하면 혼란이 온다. 이걸 "이 코드가 실제로 어디서 실행되는가"를 기준으로 나누면 훨씬 명확해진다.

```
[ 브라우저 (프론트엔드) ]           [ 서버 (백엔드) ]
   HTML / CSS / JS                    Node.js, Java, C...
   React (여기 포함)                   Express, Spring...
        └────────── HTTP로 통신 ──────────┘
```

**Q. React는 새로운 언어인가?**

아니다. React는 JS와 별개의 언어가 아니라 **JS로 화면을 더 편하게 짜도록 도와주는 라이브러리**다. `.jsx` 파일도 Vite 같은 도구가 빌드하면 결국 평범한 HTML/CSS/JS로 변환돼서 브라우저에 전달된다. 즉 실행 위치는 순수 HTML/CSS/JS와 똑같이 브라우저이고, 달라지는 건 "화면을 컴포넌트 단위 함수로 짜고 React가 알아서 갱신해준다"는 짜는 방식뿐이다.

**Q. Node.js는 브라우저의 JS와 뭐가 다른가?**

문법(JS)은 똑같이 재활용하지만 **실행되는 위치가 브라우저에서 서버 컴퓨터로 바뀐다.** 브라우저의 JS는 `document`, `window`처럼 화면을 조작하는 게 주 목적이고, Node.js는 그런 게 없는 대신 요청을 받아 DB를 조회하고 데이터를 처리해서 응답하는 게 주 목적이다. Express는 이 Node.js 위에서 웹 서버(라우팅 등)를 쉽게 만들도록 도와주는 프레임워크다.

**Q. Java와 C는 왜 React/Node.js 흐름과 이어지지 않을까?**

Java와 C는 **JS 계열이 아예 아니다.** Node.js처럼 "JS를 서버로 옮긴 것"이 아니라, 처음부터 문법도 실행 환경도 다른 별개의 언어다. Java는 JVM(자바 가상머신) 위에서 돌고 주로 대기업 백엔드 시스템·안드로이드 앱에 쓰이며, C는 컴파일러가 기계어로 직접 변환해서 운영체제·임베디드 기기처럼 하드웨어에 가까운 프로그램을 만들 때 쓴다. 웹 프론트엔드(HTML/CSS/JS → React)와는 아예 다른 커리어 트랙(백엔드/시스템 프로그래밍)에서 쓰이는 언어라, 자연스럽게 이어지지 않는 게 정상이다.

**Q. TypeScript는 이 지도에서 어디에 들어가나?**

TypeScript는 새로운 실행 위치가 아니라, **JS 위에 "타입 검사"라는 한 겹을 더 씌운 것**이다. 브라우저에서도 Node.js에서도 직접 실행되지 않고, 반드시 컴파일러(`tsc`)를 거쳐 평범한 JS로 변환된 뒤에야 실행된다. React와 조합하면 `.tsx`, Node.js와 조합하면 `.ts`가 된다. 즉 "어디서 실행되는가"의 축이 아니라 "타입을 검사하느냐"라는 별도의 축이다.

```
TypeScript(.ts/.tsx) → (tsc 컴파일, 타입 제거) → 순수 JavaScript → 브라우저 or Node.js
```

**Q. Swift, PL/pgSQL, Shell은 또 어떤 위치인가?**

이 셋은 앞서 나온 것들과도 또 다른 위치에서 실행된다.

- **Swift**: 브라우저와 무관하게, 애플 기기(iOS/macOS) 안에서 직접 실행되는 **네이티브 앱** 언어. 앱스토어에서 다운받아 설치하는 방식이라 "URL 접속 즉시 실행"되는 웹과는 배포 방식부터 다르다.
- **PL/pgSQL**: 서버도 아니고 **데이터베이스(PostgreSQL) 내부**에서 실행되는 언어. 평소 쓰는 SQL의 확장판으로, 여러 서버가 동시에 DB를 건드릴 때 정합성이 중요한 로직(재고 차감 등)을 DB 안에 직접 심어둘 때 쓴다.
- **Shell(bash/zsh/PowerShell)**: 브라우저도 서버도 DB도 아닌 **운영체제 자체를 조작**하는 언어. `npm run dev`, `git push`처럼 터미널에 치는 명령어들이 전부 셸 명령이고, 이런 명령어 여러 개를 파일 하나로 묶어 자동화(배포, 백업 등)할 때 스크립트로 쓴다.

## 개념 요약

### 전체 지도 — 실행 위치 기준

| 위치 | 기술 | 정체 |
|---|---|---|
| 브라우저 | HTML/CSS/JS | 화면의 뼈대·디자인·동작 |
| 브라우저 | React | JS로 화면을 컴포넌트 단위로 짜는 라이브러리 |
| 브라우저/서버 공통 | TypeScript | JS/Node 위에 타입 검사를 얹는 컴파일 도구 |
| 서버 | Node.js + Express | JS로 만드는 웹 서버 |
| 서버 | Java | JVM 위에서 도는 별도 언어, 대기업 백엔드/안드로이드 |
| 시스템 | C | 운영체제·임베디드 등 하드웨어에 가까운 프로그램 |
| 기기 자체(iOS) | Swift | 앱스토어에 올라가는 네이티브 앱 |
| 데이터베이스 내부 | PL/pgSQL | DB 안에서 도는 로직(트리거·함수) |
| 컴퓨터/인프라 | Shell | 명령어 자동화, 배포, 서버 관리 |

**실무 예시**: 쇼핑몰 서비스 하나를 예로 들면 — 사용자가 보는 화면은 React(브라우저), 주문 요청을 받는 서버는 Node.js/Express, 재고 차감처럼 정합성이 중요한 로직은 DB 안의 PL/pgSQL 함수, 서버 배포 자동화는 Shell 스크립트, 만약 iOS 전용 앱도 함께 낸다면 그건 Swift로 별도 제작 — 이렇게 한 서비스 안에 여러 언어가 각자 자기 위치를 맡아 동시에 존재한다.

### React·TypeScript는 "다음 언어"가 아니라 "같은 언어 위의 도구"

| 원본 | 도구를 얹은 버전 | 관계 |
|---|---|---|
| JS | React | 화면을 컴포넌트 단위로 짜는 라이브러리 (실행 위치 동일: 브라우저) |
| JS/Node.js | TypeScript | 타입 검사를 더하는 컴파일 단계 (실행 전에 JS로 변환됨) |

**실무 예시**: "React를 배웠으니 이제 새 언어를 또 배워야 하나?"라고 생각하기 쉽지만, 사실 React·TypeScript는 이미 아는 JS 지식 위에 쌓는 것이라 처음부터 새로 시작하는 게 아니다. 반면 Java·C·Swift·PL/pgSQL은 진짜로 문법부터 새로 배워야 하는 별개의 언어다.
