# trychroma.com 분석 — 우리가 배운 HTML/CSS로 실전 사이트 뜯어보기

> `HTML_CSS_요약노트.md`에서 정리한 개념을 기준으로, 실제 서비스되는 사이트(https://www.trychroma.com)의 HTML/CSS를 뜯어봅니다.
> 브라우저 개발자도구가 아니라 페이지 소스(`curl`)와 실제 배포된 CSS 파일을 직접 내려받아 분석했습니다 — 여기 나오는 코드는 전부 사이트에 실제로 존재하는 코드입니다.

---

## 1. 이 사이트는 뭐 하는 곳인가?

Chroma는 **AI 개발자를 위한 오픈소스 벡터/검색 데이터베이스**입니다. `<meta name="description">`에 이렇게 써 있습니다.

```html
<meta name="description" content="Open-source search infrastructure for AI"/>
```

즉 타겟 방문자는 "예쁜 것"보다 "이 도구를 믿고 써도 되는가"를 판단하려는 **개발자**입니다. 이 점을 기억하면서 아래 구조를 봐야, 왜 이렇게 짜여 있는지 이해가 됩니다.

---

## 2. 전체 뼈대 — 우리가 배운 `<div>`/시맨틱 태그와 비교

우리는 `header / nav / main / section / article / aside / footer`를 "이름만 봐도 어떤 구역인지 아는 상자"라고 배웠습니다(`02_html/28_index.html`). 실제 페이지에서 태그별 개수를 세어보면:

| 태그 | 개수 | 비고 |
|---|---|---|
| `<header>` | 0 | **없음** |
| `<nav>` | 3 | 상단 내비게이션 1개 + 내부 메뉴 컴포넌트 2개 |
| `<main>` | 1 | 페이지 전체를 감싸는 최상위 컨테이너 |
| `<section>` | 1 | 알고 보면 **토스트 알림 영역**(`aria-live="polite"`)이지 본문 섹션이 아님 |
| `<article>` | 0 | 없음 |
| `<aside>` | 0 | 없음 |
| `<footer>` | 0 | **없음** (링크 4열 구성의 하단 영역은 그냥 `<div>`) |
| `<h1>` | 1 | "Open-source search infrastructure for AI" |
| `<h2>`/`<h3>` | 3 / 4 | 소제목 계층 |

실제 최상단 구조:

```html
<body class="... bg-stone-50 text-foreground antialiased overflow-x-hidden">
  <main class="marketing-root relative overflow-x-hidden w-full min-h-screen grid grid-rows-[auto_1fr]">
    <div class="mb-62 z-10 flex-auto grid grid-cols-[1fr_min(80rem,100%-1rem)_1fr] ...">
      <nav class="col-2 w-full border-b lg:border-0"> ... </nav>
      <!-- 본문 내용도 전부 div -->
    </div>
  </main>
</body>
```

그리고 하단 링크 영역(우리가 흔히 "footer"라고 부를 부분)은 진짜로는:

```html
<h3 class="font-semibold text-sm mb-2">Product</h3>
<a href="/products/chromadb">Database</a>
...
<h3 class="font-semibold text-sm mb-2">Legal</h3>
<a href="/website-privacy">Privacy</a>
<div class="text-xs text-gray-600">© 2026</div>
```

**피드백**: `28_index.html`에서 배운 "의미 있는 태그를 쓰자"는 원칙과는 다르게, 이 사이트는 `header`/`footer`/`article`/`aside`를 아예 안 씁니다. 이건 나쁜 코드라기보다, **Next.js + 컴포넌트 라이브러리(Base UI/Radix 계열) 기반 실무 프로젝트에서 실제로 자주 보이는 트레이드오프**입니다. `data-slot="navigation-menu"`, `aria-expanded`, `role`류 속성으로 동작과 접근성을 보완하고, 문서 시맨틱은 포기하는 방식이죠. 다만 `<footer>`가 없으면 스크린리더 사용자가 "여기부터 꼬리말이다"라고 자동으로 인식(`contentinfo` 랜드마크)하지 못하므로, **엄밀히는 접근성 관점에서 아쉬운 지점**입니다. 반면 `h1 → h2 → h3` 제목 계층은 순서를 지켜서 잘 짜여 있습니다.

## 3. 레이아웃 — Grid를 우리보다 한 단계 더 쓴다

우리는 `03_css/13`에서 `grid-template-columns: repeat(2, 1fr)`처럼 "똑같은 너비로 n등분"하는 그리드만 다뤘습니다. 이 사이트는 이렇게 씁니다.

```css
grid-template-rows: auto 1fr;
grid-template-columns: 1fr min(80rem, 100% - 1rem) 1fr;
```

`1fr [본문] 1fr` 구조로 **배경은 화면 끝까지 꽉 채우면서, 실제 내용물은 가운데 최대 80rem(=1280px)로 제한**하는 기법입니다. 가운데 트랙을 `min()` 함수로 "80rem 아니면 화면폭-1rem 중 작은 값"으로 정해서, 큰 화면에서도 좁은 화면에서도 여백이 자연스럽게 맞춰집니다. 내용물은 `col-2`(가운데 트랙) 클래스만 붙이면 자동으로 중앙 정렬되는 셈이죠. `repeat()`만 배운 우리 입장에서는 "그리드 트랙 크기 자체를 함수로 계산한다"는 게 다음 단계로 배우면 좋은 실전 패턴입니다.

## 4. CSS 작성 방식 — 개별 `main.css` vs 유틸리티 클래스(Tailwind)

우리는 `03_css/06`에서 외부 스타일시트/내부 스타일/인라인 스타일 3가지를 배우고, 클래스 하나에 여러 속성을 몰아 쓰는 방식(`.card { ... }`)을 연습했습니다. 이 사이트는 클래스 이름 자체가 CSS 속성인 **Tailwind 유틸리티 클래스** 방식입니다.

```html
<div class="hidden lg:flex lg:items-center gap-6 lg:gap-2 ml-2 grow">
```

`hidden`(기본은 숨김) → `lg:flex`(큰 화면에서만 flex) → `lg:items-center`(큰 화면에서 세로 중앙정렬) → `gap-6`(간격) 처럼, **하나의 클래스 = 하나의 CSS 선언**입니다. 우리가 배운 "선택자 하나에 속성 여러 개"와 정반대 철학이지만, 결과적으로 만들어지는 CSS(`display:flex; align-items:center; gap:...`)는 똑같습니다. 실무에서는 이렇게 쓰는 이유가, `.card`, `.box` 같은 이름을 매번 새로 짓지 않고 재사용할 수 있기 때문입니다.

## 5. CSS 변수 — 우리가 배운 `:root`가 실제로는 "디자인 시스템"으로 쓰인다

`03_css/15`에서 배운 `:root { --main-color: red; }` + `var()` 개념이, 실제로는 이렇게 대규모 색상 팔레트로 쓰이고 있었습니다.

```css
:root {
  --background: #fff;
  --foreground: #0a0a0a;
  --primary: #171717;
  --secondary: #f5f5f5;
  --border: #e5e5e5;
  --ring: #a1a1a1;
  --radius: .625rem;
  --chart-1: #f05100;
  /* ... */
}
```

`--background`, `--foreground`, `--primary`, `--border`처럼 **색의 의미(역할)로 이름을 붙인 "디자인 토큰"** 방식입니다. 이렇게 해두면 다크모드를 켤 때 `--background` 값 하나만 바꿔도 사이트 전체 색이 따라 바뀝니다(실제로 이 마케팅 페이지의 `<html>`에는 다크 클래스가 안 붙어 있어서 라이트 테마 고정이지만, 로그인/대시보드 쪽에서 다크모드를 쓰기 위한 토큰이 이미 준비돼 있는 것으로 보입니다). `--radius: .625rem` 하나로 버튼·카드·인풋의 둥근 모서리를 통일하는 것도, 우리가 각 요소마다 `border-radius`를 따로 적었던 것과 비교되는 지점입니다.

한 가지 더 흥미로운 점 — 같은 CSS 파일 안에 **같은 변수가 `lab()` 색공간으로 한 번 더 정의**되어 있습니다.

```css
:root { --background: lab(100% 0 0); --foreground: lab(2.75381% 0 0); ... }
```

이건 최신 브라우저에서 더 넓은 색 영역(와이드 갬멋 디스플레이)을 지원하기 위한 점진적 향상(progressive enhancement)입니다. 우리가 배운 `red`, `navy`, `#fff` 같은 표기법보다 훨씬 정밀한 색 지정 방식이라, 지금 단계에서는 "이런 것도 있구나" 정도로만 알아두면 충분합니다.

## 6. 반응형 — `max-width: 600px`보다 진화한 방식

`03_css/15`에서 배운 미디어쿼리:

```css
@media (max-width: 600px) { .box { flex-direction: column; } }
```

이 사이트가 실제로 쓰는 브레이크포인트:

```css
@media (min-width: 40rem)  /* sm */
@media (min-width: 48rem)  /* md */
@media (min-width: 64rem)  /* lg */
@media (min-width: 80rem)  /* xl */
@media (min-width: 96rem)  /* 2xl */
```

차이점 두 가지가 있습니다.
1. **`px` 대신 `rem`을 씀** — 사용자가 브라우저 글자 크기를 키워놓은 경우에도 그 배율에 맞춰 반응형 기준점이 같이 늘어납니다. `px`로 고정하면 이런 사용자 설정을 무시하게 되므로, `rem` 쪽이 접근성 측면에서 더 낫습니다.
2. **작은 화면 기준(`max-width`)이 아니라 큰 화면 기준(`min-width`)으로 설계** — "기본은 모바일 스타일, 화면이 커지면 조건 추가"하는 모바일 퍼스트 방식입니다.

거기다 커리큘럼에는 없던 것도 하나 있습니다.

```css
@container field-group (min-width: 28rem) { ... }
@media (hover: hover) { ... }
@media (forced-colors: active) { ... }
```

- `@container`: **화면 크기가 아니라, 그 요소를 감싸는 부모 상자의 크기**를 기준으로 스타일을 바꾸는 최신 기능(컨테이너 쿼리)입니다. 같은 컴포넌트를 좁은 사이드바에 넣든 넓은 본문에 넣든 알아서 적응하게 만들 수 있어서, 미디어쿼리보다 한 단계 더 정교합니다.
- `(hover: hover)`: 마우스가 있는 기기에서만 호버 효과를 주고, 터치 기기에서는 안 주기 위한 기능 감지 쿼리입니다. 우리가 배운 `a:hover`만 쓰면 터치 기기에서 탭했을 때 호버 스타일이 어정쩡하게 남는 문제가 생길 수 있는데, 이 쿼리로 그걸 방지합니다.
- `(forced-colors: active)`: 윈도우 고대비 모드 같은 강제 색상 모드에서도 UI가 깨지지 않도록 하는 접근성 대응입니다.

## 7. 애니메이션 — 장식이 아니라 "제품을 설명하는 모션"

`03_css/10`에서 배운 `@keyframes float` + `hover` 조합처럼, 이 사이트도 커스텀 키프레임을 씁니다. 실제로 뽑아낸 이름만 봐도 무엇을 표현하려는지 보입니다.

```
@keyframes flow-data { ... }
@keyframes flow-down { ... }
@keyframes reasoning-pulse { ... }
@keyframes blink-enterprise { ... }
@keyframes grow-line { ... }
```

"데이터가 흐른다(flow-data/flow-down)", "AI가 추론하며 맥박친다(reasoning-pulse)", "선이 자라난다(grow-line)" — 전부 **"벡터 검색/AI 인프라"라는 제품 정체성을 시각적 은유로 표현**하는 이름들입니다. 우리가 만든 `float` 애니메이션은 "둥둥 떠다니는 느낌"이라는 순수한 장식이었다면, 이쪽은 "이 회사가 뭘 하는지"를 모션으로 설명하고 있다는 점이 다릅니다. 여기에 `accordion-down/up`, `enter/exit`, `spin`, `pulse` 같은 범용 UI 애니메이션(아코디언 펼침, 모달 등장/퇴장, 로딩 스피너)도 같이 섞여 있습니다.

## 8. 눈에 띄는 접근성 디테일

- 토스트(알림) 영역을 `<section aria-live="polite" aria-atomic="false">`로 표시 — 화면에 새 알림이 뜨면 스크린리더가 자동으로 읽어줍니다.
- 모달 제목에 `<h2 data-slot="dialog-title" class="...">Hidden</h2>` 같은, **화면에는 안 보이지만 스크린리더는 읽을 수 있는 제목**이 존재합니다. 시각적으로는 숨기고 의미만 남기는 좋은 습관입니다.
- 폼 요소 클래스에 `dark:aria-invalid`, `dark:border-input` 등이 붙어 있는데, 정작 `<html>`에는 다크 모드 클래스가 없습니다 → 이건 **공용 UI 컴포넌트 라이브러리를 그대로 가져다 썼기 때문에 남아있는 흔적**으로 보입니다. 이 마케팅 페이지 자체는 라이트 테마 고정이지만, 같은 디자인 시스템을 쓰는 다른 페이지(콘솔/대시보드)에서는 다크모드가 켜질 가능성이 있다는 뜻입니다.

## 9. 종합 피드백 — 구조가 "이 사이트의 목적"에 잘 맞는가?

Chroma는 **개발자를 설득해야 하는 오픈소스 인프라 제품**입니다. 그 목적에 맞춰 구조를 평가하면:

**잘 맞는 점**
- 폰트로 `Inter`(본문) + `IBM Plex Mono`(로고/코드 느낌)를 자체 호스팅(preload)해서 씀 → 개발자 도구다운 "코드/터미널" 느낌과, 폰트 깜빡임(FOUT) 없는 빠른 로딩을 동시에 챙김.
- `--background`/`--primary` 같은 의미 기반 CSS 변수 체계 → 프로덕트가 커져도(대시보드, 문서 사이트 등) 색상 일관성을 유지하기 쉬운 구조. "인프라 회사"라는 정체성과 어울리는 체계적인 설계.
- `flow-data`, `reasoning-pulse` 같은 애니메이션 → 눈요기가 아니라 "벡터 검색이 어떻게 동작하는지"를 은유적으로 보여주는 데 씀. 장식성 모션보다 설명적 모션에 가까움.
- `rem` 기반 반응형 + `hover:hover` + `forced-colors` 대응 → 개발자 사용자층 특성(키보드/스크린리더/고대비 모드 사용 비율이 상대적으로 높음)을 고려한 접근성 투자.

**아쉬운 점**
- `header`/`footer`/`article` 같은 시맨틱 태그를 전혀 안 써서, 우리가 배운 "구역별 의미 부여" 원칙과는 거리가 있습니다. `div` + ARIA로 대체하는 방식이 컴포넌트 라이브러리 생태계에서는 흔하지만, 순수 시맨틱 HTML 관점에서는 한 단계 타협한 것입니다.
- `<section>`이 실제 본문 섹션이 아니라 토스트 알림용으로만 쓰였다는 점도, "의미에 맞는 태그를 쓴다"는 원칙엔 다소 어긋납니다.

**한 줄 요약**: 이 사이트는 "시맨틱 태그"보다 "디자인 토큰 + 유틸리티 CSS + 접근성 속성"에 투자가 몰려 있는, **대규모 프로덕트 운영에 최적화된 실전형 코드**입니다. 우리가 배운 개념(시맨틱 태그, 박스모델, flex/grid, 미디어쿼리, CSS 변수, 애니메이션)은 전부 그대로 쓰이고 있지만, 규모가 커지면서 "이름 짓기"보다 "체계와 재사용성"을 우선하는 방향으로 진화한 모습이라고 보면 됩니다.

## 10. 전반적인 디자인 분석 — "흑백 + 아날로그 도트" 느낌의 정체

사용자가 좋아한 포인트가 정확히 코드에 근거가 있어서, 그 부분만 따로 뜯어봅니다.

### (1) 흑백이지만 "완전한" 흑백이 아니다

```css
--background: #fff;      /* 순백색 아님 → 실제 배경은 bg-stone-50(#fafaf9)로 살짝 데운 흰색 */
--foreground: #0a0a0a;   /* 순검정(#000)이 아니라 살짝 톤 낮춘 검정 */
```

배경은 차가운 순백(`#fff`)이 아니라 **살짝 베이지가 도는 종이색(stone-50, `#fafaf9`)**, 글자는 순검정이 아니라 **아주 살짝 부드러운 검정(`#0a0a0a`)** 입니다. 모니터 화면의 쨍한 흑백 대비 대신, **종이에 인쇄한 잉크 느낌**에 더 가깝게 채도를 깎아놓은 것 — 이게 "차갑지 않은 흑백" 인상을 만드는 핵심입니다.

### (2) 진짜 "도트"가 텍스트로 그려져 있다

가장 결정적인 부분인데, 실제 페이지 안에 이런 텍스트가 그대로 들어 있습니다.

```
▓░  ═════════════════════════════════════  ░▓
▓░  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  ░▓
▓░
▓░  ┌────────── CHROMA VPC ─────────────┐  ░▓
```

이미지가 아니라 **`▓ ░ ═ ─ ┌ ┐` 같은 유니코드 블록/선 문자를 글자 그대로 조합해 만든 ASCII 다이어그램**입니다. 옛날 터미널 화면이나 도트 프린터 출력물에서 보던 방식 그대로죠. 그리고 이 도트/선 부분에는 `flow-dots`라는 클래스가 붙어 있고, CSS에는

```css
.flow-dots { animation: 1.5s ease-in-out infinite flow-data; }
```

가 걸려 있어서, **이 도트들이 실제로 깜빡이며 흐르는 애니메이션**이 됩니다. 즉 "인프라 안에서 데이터가 흐른다"는 개념을, 그림이 아니라 옛날 컴퓨터 텍스트 화면 스타일로 재현하고, 거기에 모션까지 준 것 — 사용자가 느낀 "아날로그 도트" 감성의 정체가 바로 이겁니다.

### (3) 배경에도 얇은 도트/줄무늬 텍스처가 깔려 있다

CSS에서 발견한 배경 이미지를 디코딩해보면, 가로로 얇은 줄이 2px 간격으로 반복되는 아주 작은(204×18px) 타일 패턴이고, 색은 순검정이 아니라 짙은 세피아톤 잉크색(`#27201C`)입니다. 화면 전체가 아니라 특정 구역의 얇은 구분선/텍스처로 반복 삽입되어, 인쇄물의 망점(halftone)이나 등사기 잉크 느낌을 은은하게 줍니다.

### (4) 카드 그림자도 "부드러운 그림자"가 아니라 "딱 떨어지는 그림자"

```css
.mac-style { border-radius: 2px; box-shadow: 6px 6px #f0f0f0; transition: all .3s; }
```

요즘 대부분의 사이트는 `box-shadow: 0 4px 12px rgba(0,0,0,.1)`처럼 **블러(퍼짐)가 있는 부드러운 그림자**를 씁니다. 여기는 블러 값이 아예 `0`이고, 그냥 오른쪽 아래로 6px씩 밀린 **딱딱한 단색 그림자**입니다. 이건 실제 물체가 아니라 종이를 오려 붙인 것 같은, 옛날 카드/스티커/인쇄물에서 보던 그림자 방식이라 "디지털 화면"보다 "손으로 만든 인쇄물" 느낌을 줍니다.

### (5) 모노스페이스 폰트를 전체 페이지에 광범위하게 사용

`font-mono` 클래스가 페이지 전체에서 185번 등장합니다. 본문용 `Inter`(가변폭 폰트) 위에, 코드/라벨/다이어그램 부분마다 고정폭 폰트(`IBM Plex Mono`)를 계속 겹쳐 씀 → 타자기·터미널·영수증 프린터에서 보던 "글자 폭이 다 똑같은" 인상을 페이지 전반에 깔아둔 것입니다.

### 종합

이 다섯 가지가 합쳐져서 "차가운 디지털 화면"이 아니라 **"오래된 컴퓨터 터미널 + 인쇄물"이 섞인 아날로그 감성**이 만들어집니다.

| 요소 | 만약 흔한 방식이었다면 | 실제로는 |
|---|---|---|
| 배경/글자색 | 순백(`#fff`) / 순검정(`#000`) | 살짝 데운 흰색 / 살짝 눌린 검정 |
| 데이터 흐름 표현 | 아이콘·일러스트 이미지 | 유니코드 도트/선 문자 + 애니메이션 |
| 구분선 텍스처 | 단색 `border` | 얇은 반복 도트 타일 이미지 |
| 그림자 | 블러 있는 부드러운 그림자 | 블러 0, 딱딱한 오프셋 그림자 |
| 폰트 | 가변폭 본문 폰트만 | 본문 폰트 + 모노스페이스 폰트 이중 사용 |

이 조합은 Chroma가 "AI 인프라/데이터베이스"라는, 원래는 굉장히 눈에 안 보이고 추상적인 제품을 "옛날 서버실 터미널 화면을 들여다보는 느낌"으로 시각화한 것이라고 볼 수 있습니다. 화려한 최신 그래픽보다 **"진짜 엔지니어가 쓸 법한 화면"이라는 신뢰감**을 주는 방향으로 색, 텍스처, 그림자, 폰트를 전부 일관되게 맞춘 셈이라, 사용자가 느낀 "흑백 + 아날로그 도트" 인상은 우연이 아니라 꽤 의도적인 디자인 언어입니다.
