# [node 02] 동적 라우트 파라미터와 미들웨어

> 작성: 2026-07-14

## Q&A

**Q. `app.get('/api/users', ...)`만 있는 서버에서 `localhost:3000`(루트 주소)으로 접속하면 왜 아무것도 안 나오나?**

`app.get('/api/users', ...)`는 정확히 `/api/users`라는 주소로 요청이 왔을 때만 반응한다. 루트 주소(`/`)에 대한 라우트가 따로 없으면 Express는 "Cannot GET /"라는 기본 404를 보여준다. 서버 안에 데이터가 없는 게 아니라, 그 데이터를 꺼내는 **정확한 경로로 접속하지 않은 것**이 원인이다.

```js
app.get('/api/users', (req, res) => {
  res.json([{ id: 1, name: 'LEE' }])
})
// localhost:3000        → Cannot GET / (라우트 없음)
// localhost:3000/api/users → 정상 응답
```

**Q. `app.listen(3000, () => console.log("http://localhost:3000"))`에서 뒤에 붙는 콘솔 로그는 왜 있는 건가?**

`app.listen`의 첫 번째 인자(`3000`)는 실제로 서버가 대기할 포트 번호라 필수지만, 두 번째 콜백 함수는 **서버가 잘 열렸는지 사람이 눈으로 확인하려고 찍는 안내 로그일 뿐** 기능적으로는 없어도 서버가 똑같이 동작한다. 다만 이 문자열은 앞의 포트 번호와 실제로 일치하게 직접 맞춰 적어야 한다(자동으로 맞춰주지 않음).

**Q. 주소 중간에 바뀌는 값(예: 특정 유저 하나만 조회)은 어떻게 처리하나?**

주소에 `:이름` 형태로 자리를 만들면, 그 자리에 들어온 실제 값을 `req.params.이름`으로 꺼낼 수 있다. 예를 들어 `/api/users/3`으로 요청이 오면 `req.params.id`에 문자열 `"3"`이 담긴다.

```js
const users = [{ id: 1, name: '지니' }, { id: 2, name: '철수' }, { id: 3, name: '영희' }]

app.get('/api/users/:id', (req, res) => {
  const user = users.find(u => u.id === Number(req.params.id))  // params는 항상 문자열이라 Number 변환 필수
  if (!user) return res.status(404).json({ error: '없는 유저' })
  res.json(user)
})
```

**Q. 위 코드에서 `users` 배열 선언을 깜빡하면 어떤 에러가 나나?**

`users.find(...)`를 호출하는 시점에 `users`라는 이름 자체가 어디에도 선언돼 있지 않으면 `ReferenceError: users is not defined`가 나면서 그 요청 처리가 그대로 죽는다. 라우트 로직을 짜기 전에 **검색할 대상 데이터가 먼저 존재해야 한다**는 당연하지만 자주 놓치는 순서다.

**Q. 라우트 안에서 `res.json(...)`을 조건 분기보다 먼저 호출하면 무슨 문제가 생기나?**

응답은 한 요청당 한 번만 보낼 수 있는 게 원칙이라, 조건(404 체크 등)을 확인하기도 전에 무조건 먼저 응답을 보내버리면 그 뒤에 있는 분기 로직들은 이미 응답이 나간 뒤라 사실상 의미가 없어진다(실행은 되지만 클라이언트에 반영이 안 되거나 헤더 재전송 에러로 이어질 수 있음). **"찾기 → 없으면 404 → 있으면 응답"** 순서를 반드시 지켜야 한다.

**Q. PUT으로 body에 담아 보낸 데이터가 서버에서 `req.body.name`으로 꺼내려는데 `req.body` 자체가 `undefined`로 나온다면?**

`app.use(express.json())` 미들웨어가 빠진 것이다. 이게 없으면 클라이언트가 보낸 JSON 문자열을 Express가 자동으로 객체로 풀어주지 않아서 `req.body`가 아예 `undefined`가 되고, 그 상태에서 `.name`을 읽으려다 서버 내부 에러(500)가 나며 기본 HTML 에러 페이지가 응답된다.

```js
app.use(express.json())  // 必 — 이게 없으면 req.body는 항상 undefined

app.put('/api/users/:id', (req, res) => {
  const u = users.find(u => u.id === Number(req.params.id))
  if (!u) return res.status(404).json({ error: '없는 유저' })
  u.name = req.body.name
  res.json(u)
})
```

**Q. 미들웨어 안에서 로그만 찍고 요청이 계속 멈춰있다면?**

미들웨어는 `next()`를 명시적으로 호출해야 다음 단계(다음 미들웨어 또는 최종 라우트)로 넘어간다. `next()` 호출이 빠지면 요청 처리가 그 미들웨어에서 영원히 멈추고, 클라이언트는 응답을 못 받아 계속 로딩 상태가 된다.

```js
// 잘못된 예 — next() 없음 → 이 뒤로 요청이 영원히 안 넘어감
app.use((req, res, next) => {
  console.log(req.method, req.url)
})

// 올바른 예
app.use((req, res, next) => {
  console.log(req.method, req.url)
  next()  // 必 — 이게 있어야 다음 라우트로 진행됨
})
```

## 개념 요약

### 라우트 매칭은 "정확한 주소"에만 반응

| 요청 주소 | 등록된 라우트 | 결과 |
|---|---|---|
| `/api/users` | `app.get('/api/users', ...)` | 정상 응답 |
| `/` (루트) | 등록 안 됨 | Cannot GET / |

**실무 예시**: 프론트엔드 개발자가 "API가 안 되는데요?"라고 할 때, 실제로는 서버가 죽은 게 아니라 브라우저 주소창에 API 경로 없이 도메인만 쳐서 생기는 착각인 경우가 흔하다.

### 동적 파라미터 `:id`와 `req.params`

| 항목 | 내용 |
|---|---|
| 선언 | `app.get('/api/users/:id', ...)` |
| 값 꺼내기 | `req.params.id` |
| 타입 | 항상 **문자열** — 숫자 비교 전에 `Number()` 필요 |

**실무 예시**: 게시판의 `/posts/:postId`, 쇼핑몰의 `/products/:productId`처럼 "목록 중 하나만 콕 집어 조회/수정/삭제"하는 거의 모든 라우트가 이 패턴을 쓴다.

### 요청 처리 순서 원칙

```js
app.get('/api/users/:id', (req, res) => {
  const user = users.find(u => u.id === Number(req.params.id))
  if (!user) return res.status(404).json({ error: '없는 유저' })  // ① 먼저 체크
  res.json(user)                                                  // ② 그 다음 응답
})
```

**실무 예시**: "찾기 → 없으면 에러 응답하고 함수 종료(`return`) → 있으면 정상 응답"은 API 라우트 작성의 기본 골격이라, 이 순서가 뒤바뀌면 이후 어떤 CRUD 기능을 만들어도 같은 실수가 반복된다.

### `express.json()` 미들웨어

| 상황 | `express.json()` 없음 | 있음 |
|---|---|---|
| 클라이언트가 보낸 JSON body | `req.body === undefined` | 자동으로 객체로 파싱됨 |

**실무 예시**: POST/PUT처럼 body를 받는 라우트를 만들 때 이 한 줄을 빼먹는 실수가 매우 흔하다. "분명 프론트에서 다 보냈는데 서버에서 안 받아진다"는 버그의 단골 원인.

### 미들웨어와 `next()`

| 역할 | 설명 |
|---|---|
| `app.use((req,res,next)=>{...})` | 모든 요청이 최종 라우트로 가기 전에 거치는 공용 처리 지점 |
| `next()` | 다음 단계로 넘기는 신호 — 없으면 요청이 그 자리에서 멈춤 |

**실무 예시**: 로깅, 인증 확인(로그인 여부), CORS 처리 등은 전부 미들웨어로 구현하는데, 이 중 하나라도 `next()`를 안 부르면 그 뒤에 있는 모든 라우트가 먹통이 되는 치명적인 실수로 이어진다.
