## 폴더별 코드로 보는 학습 흐름

`11_python/07~17` 폴더의 `.py` 파일을 순서대로 읽었을 때 드러나는 학습 흐름 정리.

**07 — `py_zip.py`**: `zip(names, scores)`로 두 리스트를 한 번에 순회하는 기초 문법. 리스트 두 개를 인덱스 기준으로 나란히 묶어서 `for`로 도는 패턴의 첫 예시.

**10 — `py_figlet_termcolor.py`**: `pyfiglet`(큰 글씨 아스키아트) + `termcolor`(색상)를 조합해서 문자열을 꾸며 출력하는 예제. 외부 패키지를 두 개 동시에 쓰는 첫 사례라, 여기서부터 "설치는 됐는데 실행이 안 되는" 가상환경 이슈가 등장함.

**11 — `py_def.py` / `py_def_kg2p.py` / `py_figlet.py`**: 세 파일 모두 `def 함수명(파라미터): ... print(...)` 구조의 기본 함수 정의 연습. 단위 변환(미터→피트, kg→pound) 두 개는 "계산 후 바로 출력"하는 가장 단순한 형태이고, `py_figlet.py`의 `good_sentence()`는 타입힌트(`text: str`)와 docstring, 그리고 `pyfiglet` + `colored`를 함수 안에서 조합하는 좀 더 발전된 형태.

**12 — `py_return.py`**: `highlight()` 함수가 `return color_text`로 값을 돌려주는 구조. 처음엔 반환값을 그냥 버리는 코드였다가, 지금은 `results = highlight(...)`로 받아서 `print(results)`로 출력하도록 수정되어 있음 — `return`(값 전달)과 `print`(화면 출력)이 별개의 동작이라는 걸 보여주는 파일.

**16 — `py_class.py`**: `class World`로 생성자(`__init__`)와 속성(`self.name`, `self.population`), 메서드(`hello()`, `info()`)를 정의하고 `World("Korea", 51000000)`로 객체를 만들어 메서드 두 개를 호출. 함수 여러 개를 하나의 "틀" 안에 묶어서 관련 데이터(`name`, `population`)와 동작(`hello`, `info`)을 함께 관리하는 구조로 넘어가는 파일.

**17 — `py_lambda.py`**: `def`로 만들던 함수를 `lambda`로 압축하는 연습이 3단계로 쌓여 있음.
- `sentence = lambda sentence: print(colored(pyfiglet.figlet_format(sentence), "blue", "on_green"))` — 중간 변수 없이 함수 호출을 겹쳐서 한 줄로 압축
- `st = lambda st: print(pyfiglet.figlet_format(st))` → `st("hello")`로 호출 (바깥에 `print()`를 또 씌우지 않음 — `None` 이중 출력 문제를 피한 형태)
- `decorate_text = lambda text: pyfiglet.figlet_format(text)` → 이건 `print()`가 아니라 **값을 반환하는** lambda라서 `print(decorate_text("LAMBDA"))`로 바깥에서 출력. 같은 폴더 안에서 "lambda가 직접 출력하는 경우"와 "lambda가 값만 반환하고 호출부에서 출력하는 경우"를 나란히 비교해볼 수 있음.

## 개념 요약

| 파일 | 핵심 포인트 |
|---|---|
| `07/py_zip.py` | 두 리스트를 인덱스 기준으로 동시 순회 |
| `10/py_figlet_termcolor.py` | 외부 패키지 2개 조합 + 가상환경 의존성 노출 |
| `11/py_def.py`, `py_def_kg2p.py` | 계산 → 즉시 출력하는 최소 단위 함수 |
| `11/py_figlet.py` | 타입힌트·docstring이 붙은, 조금 더 정돈된 함수 |
| `12/py_return.py` | `return`과 `print`를 분리해서 값을 받고(`results =`) 출력하는 흐름 |
| `16/py_class.py` | 속성+메서드를 묶는 `class`, `__init__`으로 초기화 |
| `17/py_lambda.py` | `def` → `lambda` 변환, "출력하는 lambda" vs "값만 반환하는 lambda" 구분 |

```python
# zip: 길이 다를 때
names = ["뽀삐", "초코", "쿠키"]
scores = [95, 88]
for name, score in zip(names, scores):
    print(f"{name}: {score}점")
# 뽀삐: 95점 / 초코: 88점만 출력, 쿠키는 버려짐

# return vs print
def highlight(text: str, color: str) -> str:
    return colored(text, color)

results = highlight("good", "yellow")
print(results)

# lambda 두 스타일: 출력형 vs 반환형
sentence = lambda sentence: print(colored(pyfiglet.figlet_format(sentence), "blue", "on_green"))
decorate_text = lambda text: pyfiglet.figlet_format(text)
print(decorate_text("LAMBDA"))
```

**실무 예시**: `py_return.py`가 보여주듯, 실무 코드에서는 함수가 값을 반환(`return`)하게 만들어야 그 결과를 로그로 남기거나, 다른 함수에 다시 넘기거나, 테스트로 검증할 수 있음 — 함수 안에서 바로 `print`만 하면 그 값을 재사용할 방법이 없어짐. `py_lambda.py`의 "출력 lambda vs 반환 lambda" 구분도 마찬가지로, `map()`/`sorted()`의 `key=` 인자처럼 **값을 다음 단계로 넘겨야 하는 자리**에는 반드시 반환형 lambda를 써야 하고, 사이드이펙트(출력, 로깅 등)가 목적일 때만 `print`를 본문에 넣는 식으로 구분해서 씀. `zip()`은 데이터베이스에서 뽑은 "컬럼명 리스트"와 "값 리스트"를 한 줄씩 짝지어 딕셔너리로 만들 때(`dict(zip(columns, row))`) 자주 쓰이고, 가상환경 인터프리터 불일치는 "내 컴퓨터에선 되는데 팀원 컴퓨터/서버에선 안 된다"는 문제의 단골 원인이라 항상 어떤 인터프리터가 활성화됐는지 확인하는 습관이 필요함.
