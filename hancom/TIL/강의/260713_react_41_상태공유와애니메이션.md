## Q&A

**Q. 컴포넌트에 여러 개의 `useState`가 있을 때, 실제로 이걸 몇 개나 어떤 용도로 쓰게 되나?**

오늘 만든 칵테일 앱(`my-app2`) 기준으로 보면 `useState`는 크게 세 가지 용도로 쓰인다.

| 용도 | 예시 |
|---|---|
| API로 받아온 데이터 저장 | `const [cocktails, setCocktails] = useState([])` |
| 화면에 뭘 보여줄지 결정하는 상태 | `const [isLoading, setIsLoading] = useState(true)`, `const [phase, setPhase] = useState('idle')` |
| 사용자 상호작용 추적 | `const [hovered, setHovered] = useState(null)` |

한 컴포넌트 안에 state가 3개 이상 얽히기 시작하면(`cards`, `picked`, `phase`처럼) 각각을 따로 `useState`로 관리하는 대신 `useReducer`로 묶는 게 다음 단계로 넘어가는 자연스러운 방향이다.

**Q. A 페이지에서 B 페이지로 이동할 때 정보를 넘기고 싶은데, `react-router-dom`의 `Link state`를 쓰면 안 되는 경우가 있나?**

있다. `<Link to="/" state={{ from: 'whisky' }}>`처럼 넘긴 `state`는 브라우저의 히스토리 엔트리에 저장된다. 그런데 이 히스토리 state는 **새로고침해도 사라지지 않는다.** "새로고침하면 무조건 초기 상태로 리셋되길" 원하는 로직(예: "레시피 페이지에서 돌아왔을 때만 추천 문구를 보여주고, 그냥 새로고침하면 항상 기본 문구를 보여주기")에는 맞지 않는다.

해결책은 라우터 state 대신 **순수 JS 모듈 변수**를 쓰는 것이다.

```js
// navMemory.js
let lastSpirit = null
export const setLastSpirit = (spirit) => { lastSpirit = spirit }
export const getLastSpirit = () => lastSpirit
```

이 변수는 SPA 안에서 페이지를 이동하는 동안에는(JS 런타임이 계속 살아있으므로) 값이 유지되지만, **새로고침하면 스크립트가 처음부터 다시 실행되면서 자동으로 `null`로 초기화**된다. "SPA 내 이동에서는 유지, 새로고침에서는 리셋"이라는, 라우터 state와 정반대의 특성이 필요할 때 쓰는 패턴이다.

**Q. 카드가 화면의 원래 자리에서 화면 중앙으로 자연스럽게 확대 이동하는 애니메이션은 어떻게 만드나?**

**FLIP 기법**(First-Last-Invert-Play)의 축약 버전을 썼다. 핵심은 애니메이션을 시작하기 **직전의 실제 화면 좌표를 `getBoundingClientRect()`로 먼저 재두는 것**이다.

```jsx
const handlePick = (card, event) => {
  const rect = event.currentTarget.getBoundingClientRect()
  setPicked({ ...card, rect }) // 클릭 시점의 실제 위치를 기억
}

// 렌더링
<div
  className={isRevealed ? 'draw-card--floating is-revealed' : 'draw-card--floating'}
  style={isRevealed ? undefined : { top: picked.rect.top, left: picked.rect.left, width: picked.rect.width, height: picked.rect.height }}
/>
```

처음엔 인라인 스타일로 "원래 있던 자리 그대로"에 카드를 띄우고, 그다음 프레임에 `isRevealed` 클래스를 붙여 CSS(`top: 50%; left: 50%; width: ...; transform: translate(-50%, -50%)`)가 적용되게 하면, `transition: top 0.7s, left 0.7s, width 0.7s ...`가 알아서 "원래 위치 → 중앙"으로 부드럽게 보간해준다. 카드가 몇 번째 줄, 몇 번째 칸에 있었는지 미리 계산할 필요 없이 **어디서 클릭하든 항상 그 자리에서 자연스럽게 움직인다.**

**Q. 카드가 3D로 뒤집히는 효과는 순수 CSS로 되나?**

된다. 세 가지 속성 조합이 핵심이다.

```css
.card {
  perspective: 700px; /* 부모: 원근감(입체감) 정도 */
}
.card__inner {
  transform-style: preserve-3d; /* 자식들의 3D 배치를 유지 */
  transition: transform 0.6s ease;
}
.card__inner.is-flipped {
  transform: rotateY(180deg);
}
.card__face {
  position: absolute;
  inset: 0;
  backface-visibility: hidden; /* 뒷면을 봤을 때는 숨김 */
}
.card__back {
  transform: rotateY(180deg); /* 뒷면은 미리 180도 돌려둠 */
}
```

앞면·뒷면을 같은 자리에 겹쳐두고 뒷면만 미리 180도 돌려두면, 부모(`.card__inner`)가 `rotateY(180deg)`로 회전할 때 앞면은 사라지고(`backface-visibility: hidden`) 뒷면이 정방향으로 나타나는 것처럼 보인다.

**Q. `isLoading`을 API 응답 즉시 꺼버리면 로딩 애니메이션이 너무 빨리 사라지는데, 최소 시간을 보장하려면?**

API 요청과 "최소 대기 시간"을 `Promise.all`로 묶으면 된다.

```js
const wait = (ms) => new Promise((resolve) => setTimeout(resolve, ms))

Promise.all([
  fetchCocktail(name),
  wait(1500), // 최소 1.5초
]).then(([drink]) => setIsLoading(false))
```

`Promise.all`은 **가장 오래 걸리는 것**이 끝나야 완료되므로, API가 200ms만에 응답해도 최소 1.5초는 로딩 화면이 유지되고, API가 3초 걸리면 3초를 그대로 기다린다.

## 개념 요약

### 라우터 state vs 순수 JS 변수

| | 지속 범위 | 새로고침 시 |
|---|---|---|
| `<Link state={...}>` | 브라우저 히스토리 엔트리 | **유지됨** |
| 모듈 스코프 `let` 변수 | 현재 JS 런타임(탭이 살아있는 동안) | **초기화됨** |

**실무 예시**: "직전에 어떤 페이지에서 왔는지"에 따라 다음 화면의 안내 문구를 바꾸되, 사용자가 그냥 새로고침했을 때는 원래 상태로 돌아가야 하는 온보딩 플로우, 설문 진행 상태 등에 이 구분이 중요하다.

### FLIP 애니메이션

```
1. First : 애니메이션 시작 전 요소의 실제 위치를 getBoundingClientRect()로 측정
2. Last  : 목표 상태(클래스 등)를 적용해 최종 위치/크기를 결정
3. Invert: (본격 FLIP은 최종-시작 차이만큼 역보정 후 transition으로 되돌리지만,
           여기서는 "시작 위치를 인라인 스타일로 고정 → 다음 틱에 목표 클래스 적용"으로 단순화)
4. Play  : CSS transition이 나머지를 부드럽게 처리
```

**실무 예시**: 장바구니에 상품을 담을 때 상품 이미지가 리스트에서 장바구니 아이콘으로 날아가는 연출, 칸반 보드에서 카드를 다른 컬럼으로 옮길 때의 부드러운 이동 등이 전부 이 기법을 쓴다.

### 최소 로딩 시간 보장

```js
Promise.all([실제요청, wait(최소시간)])
```

**실무 예시**: 결제 처리 API가 너무 빨리 끝나면 오히려 "진짜 처리된 게 맞나?" 불안감을 주므로, 로딩 스피너를 최소 1~2초는 보여주는 결제/저장 플로우에 자주 쓰는 트릭이다.

### CSS 3D 카드 뒤집기

| 속성 | 역할 |
|---|---|
| `perspective` | 부모에 원근감 부여 (숫자가 작을수록 입체감 과장) |
| `transform-style: preserve-3d` | 자식 요소들의 3D 배치를 평면화하지 않고 유지 |
| `backface-visibility: hidden` | 회전해서 뒤로 넘어간 면은 안 보이게 |
| `rotateY(180deg)` | 실제 회전 애니메이션 |

**실무 예시**: 플래시카드 학습 앱, 카드 뒤집기형 메모리 게임, 상품 카드 앞(사진)/뒤(상세 스펙) 토글 UI 등에서 표준적으로 쓰이는 구현 방식이다.
