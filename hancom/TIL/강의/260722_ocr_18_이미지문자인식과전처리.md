# [ocr 18] Tesseract 이미지 문자 인식과 전처리 트러블슈팅

> 작성: 2026-07-22

## Q&A

**Q. pytesseract를 설치했는데도 `ModuleNotFoundError: No module named 'pytesseract'`가 나는 이유는?**

패키지 자체는 문제없이 설치돼 있어도(예: base 콘다 환경), VSCode가 실제로 실행하는 파이썬 인터프리터가 그 환경이 아닐 수 있다. `conda env list`로 확인해보면 base 외에 nlp, ocrpy, ocrpy38, py39 같은 여러 환경이 있을 수 있는데, IDE가 base가 아닌 다른 환경(예: ocrpy38)으로 파일을 실행하면 그 환경에는 패키지가 없어서 같은 에러가 반복된다. `pip show 패키지명`으로 어느 환경에 설치돼 있는지, `python -c "import sys; print(sys.executable)"`로 지금 어느 인터프리터가 활성인지 비교해보면 원인이 바로 드러난다.

**Q. 이미지를 불러왔고 에러도 안 나는데 `image_to_string()` 결과가 빈 문자열(`''`)로만 나오는 이유는?**

pytesseract는 Tesseract 내부에서 발생하는 경고성 오류를 예외로 던지지 않고 조용히 통과시키는 경우가 있다. 실제로 tesseract.exe를 터미널에서 직접 실행해보면 `Error in boxClipToRectangle: box outside rectangle`, `Error in pixScanForForeground: invalid box` 같은 Leptonica 내부 오류가 stderr에 찍히는데, 파이썬 스크립트에서는 이게 그냥 빈 결과로만 나타난다. 원인은 사진 속에 이미지 경계에 걸쳐 지나가는 가는 선(벽의 균열 같은 것)이 있으면, Tesseract의 자동 레이아웃 분석이 이걸 텍스트 영역으로 착각해 이미지 경계를 벗어나는 바운딩 박스를 계산하면서 내부적으로 깨지기 때문이다. `tesseract 이미지파일 stdout -l eng` 처럼 CLI로 직접 실행해서 stderr를 확인하는 게 pytesseract만 봐서는 안 보이는 원인을 잡는 가장 빠른 방법이다.

**Q. `--psm 6`을 줘서 위 크래시는 피했는데, 이번엔 글자가 멀쩡히 보이는데도 완전히 의미 없는 문자열이 나온다. 왜?**

이미지 해상도가 너무 커서(예: 4000×6000) 글자 획 높이가 픽셀 기준으로 수백 px에 달하면, Tesseract의 문자 분류기가 기대하는 정상 범위(스캔 문서 기준 대략 20~40px)를 훨씬 벗어난다. Tesseract는 인식 전에 각 글자를 내부적으로 정규화된 크기 셀로 리샘플링해서 특징을 뽑는데, 원본 글자가 이 정규화 크기보다 지나치게 크면 이 과정에서 획 모양 정보가 오히려 뭉개져서 벽 질감 노이즈까지 전부 "글자"로 오인식하고 의미 없는 문자열을 뱉는다. 그레이스케일 변환 → 이진화(threshold) → 폭 800px 정도로 축소하는 전처리를 거치면 정상적으로 읽힌다. 즉 "사람 눈에 잘 보이는가"가 아니라 "Tesseract 내부 정규화 파이프라인이 기대하는 크기에 맞는가"가 핵심이다.

**Q. 반대로 스크린샷처럼 작은 이미지(342×584)에서는 한글은 잘 읽히는데 그 안에 섞인 영어 단어(`follow-up`, `feedback` 등)만 유독 깨지는 이유는?**

이번엔 해상도가 너무 작아서 생기는, 위와 정반대 문제다. 한글은 초성·중성·종성이 결합된 복잡한 글자 형태라 저해상도에서도 구분이 어느 정도 유지되지만, 영어 알파벳은 단순한 획이라 몇 px 안 되는 크기에서는 뭉개져 비슷하게 생긴 다른 글자(숫자·기호)로 오인식되기 쉽다. `Image.resize((w*3, h*3), Image.LANCZOS)`로 3배 업스케일하면 대부분의 영어 단어가 정상적으로 인식된다. 여기서 중요한 점은 LANCZOS 리샘플링이 없던 정보를 만들어내는 게 아니라(AI 초해상도와는 다름), 원본 픽셀 데이터를 부드럽게 재배치해서 Tesseract 내부 재정규화 과정이 원래 획 모양을 안정적으로 복원할 수 있게 도와주는 것뿐이라는 점이다.

**Q. Tesseract 같은 전통 OCR과 Claude/GPT 같은 Vision-Language 모델(VLM)의 근본적인 차이는?**

Tesseract는 "이 픽셀 뭉치가 어떤 글자 모양과 제일 비슷한가"를 판단하는 시각 패턴 매칭이 핵심이고, 언어별 사전(dictionary)과 n-gram 확률로 결과를 보정하는 정도가 전부다. 그래서 "이 단어가 사전에 있는 형태에 가까운가"만 통계적으로 볼 뿐, 문맥이 의미상 말이 되는지는 이해하지 못한다. 반면 VLM은 글자 모양 인식과 언어 이해를 분리하지 않고 동시에 수행하기 때문에, 픽셀상 살짝 뭉개져도 문맥으로 유추하고 "이 결과 이상한데?"를 스스로 인지할 수 있다. 다만 VLM은 이미지 1장당 비용·응답 속도가 Tesseract(로컬, 사실상 무료)보다 훨씬 크기 때문에, 실무에서는 Tesseract로 대량을 빠르게 1차 처리하고 confidence가 낮거나 애매한 부분만 VLM에 재검증시키는 하이브리드 구조를 쓴다.

## 개념 요약

| 개념 | 설명 |
|---|---|
| `pytesseract.pytesseract.tesseract_cmd` | Tesseract 실행 파일(.exe) 경로 지정 |
| `image_to_string(image, lang, config)` | 이미지에서 텍스트를 문자열로 추출 |
| `lang` 옵션 | 인식 언어 지정, `+`로 다중 언어 동시 인식(`eng+kor`) 가능 |
| `config='--psm N'` | 페이지 분할 모드(Page Segmentation Mode) 지정. 기본 자동분석이 실패/오작동할 때 강제로 방식을 바꿈 |
| Leptonica 내부 오류 | `boxClipToRectangle`, `pixScanForForeground` 등 — stderr에만 찍히고 pytesseract는 예외를 던지지 않아 결과가 조용히 빈 문자열이 됨 |
| 해상도 스위트스팟 | 글자 획 높이가 대략 20~40px 범위여야 정상 인식. 너무 크면(초고화질 사진) 정규화 과정에서 실패, 너무 작으면(저해상도 스크린샷) 획 자체가 뭉개짐 |
| 그레이스케일+이진화 | `image.convert("L")` → `point(lambda p: 255 if p > threshold else 0)`. 노이즈(배경 질감) 제거, 순수 흑백으로 대비 강화 |
| `Image.LANCZOS` | 고품질 보간 리샘플링 필터. 새 정보를 만들지 않고 기존 픽셀을 부드럽게 재배치 |
| Tesseract vs VLM | Tesseract=시각패턴+얕은 사전교정, VLM=시각+언어이해 통합(문맥 기반 자가검증 가능하나 비용·속도 부담 큼) |

```python
# 초고해상도 사진(글자가 너무 큰 경우) 전처리
gray = image.convert("L")
binarized = gray.point(lambda p: 255 if p > 150 else 0)
ratio = 800 / binarized.width
resized = binarized.resize((800, int(binarized.height * ratio)))
results = pytesseract.image_to_string(resized, lang='eng', config='--psm 6')
```

```python
# 저해상도 스크린샷(영어가 너무 작은 경우) 전처리
w, h = image.size
resized = image.resize((w * 3, h * 3), Image.LANCZOS)
results = pytesseract.image_to_string(resized, lang='eng+kor')
```

**실무 예시**: 영수증·명함 스캔 앱은 촬영 각도·해상도가 제각각이라 "이미지를 무조건 어떤 고정 크기로 정규화한 뒤 OCR에 넣는" 전처리 단계가 항상 들어간다. 카메라로 찍은 초고해상도 사진은 리사이즈+이진화, 웹에서 캡처한 저해상도 스크린샷은 업스케일 — 이렇게 원본 상태에 따라 다른 전처리를 자동 판단해서 적용하는 로직이 실제 OCR 파이프라인의 핵심 노하우다. 대량 처리(수만 건 문서 디지털화)에는 비용 때문에 Tesseract/PaddleOCR을 1차로 쓰고, 정확도가 중요한 소량 건이나 최종 검수 단계에만 VLM을 투입하는 하이브리드 구조가 일반적이다.
